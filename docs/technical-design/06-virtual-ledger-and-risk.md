# 06. 虚拟账本与风险控制设计

## 1. 设计目标

虚拟账本是 V1 的核心差异化能力。由于 OKX 单账户单向持仓只能看到真实净仓位，系统必须额外维护频道级虚拟子仓位，才能正确解释：

- 某条消息影响的是哪个频道的哪一笔交易叙事
- 多个频道同时作用于同一标的时各自还有多少虚拟持仓
- 某个频道的盈亏、挂单和生命周期如何计算

## 2. 核心对象

### 2.1 `virtual_positions`

表达某个频道的一段独立交易生命周期。

关键字段建议包括：

- `virtual_position_id`
- `channel_id`
- `symbol`
- `side`
- `status`
- `virtual_qty`
- `virtual_avg_px`
- `reserved_margin`
- `realized_pnl`
- `unrealized_pnl`

### 2.2 `virtual_position_events`

表达虚拟持仓的事件流。建议事件类型包括：

- `initiated`
- `order_planned`
- `order_submitted`
- `fill_applied`
- `add_position`
- `reduce_position`
- `take_profit`
- `stop_loss`
- `break_even_move`
- `order_canceled`
- `closed`

## 3. 生命周期设计

建议虚拟子仓位生命周期如下：

- `initiated`：决策成立，但尚未形成真实持仓
- `opening`：订单已提交，等待形成实际持仓
- `open`：已形成有效虚拟持仓
- `adjusting`：正在补仓 / 减仓 / 移动止损 / 分批止盈
- `closed`：虚拟数量归零，生命周期结束
- `canceled`：开仓计划未形成持仓即取消

## 4. 成交映射策略

### 4.1 基本原则

- 账本更新以真实成交为准，不以计划数量为准
- 一个 AI 动作可映射多笔订单，一个订单可有多笔成交
- 每笔成交落账时都应能知道“它属于哪个虚拟持仓”

### 4.2 关联方式

推荐关联链路：

`message -> decision -> execution plan -> exchange_order -> exchange_fill -> virtual_position_event`

### 4.3 偏差处理

若计划数量与实际成交数量不一致：

- 账本按真实成交更新
- 偏差写入事件说明
- 控制台可展示“计划 / 实际”差异

## 5. 盈亏计算

### 5.1 已实现盈亏

由以下事件驱动更新：

- 部分止盈成交
- 全部平仓成交
- reduce-only 平仓成交

### 5.2 未实现盈亏

由以下信息估算：

- 虚拟持仓当前数量
- 虚拟均价
- 当前市场标记价格或可替代价格

V1 中可先采用近似计算，但口径必须稳定。

## 6. 保证金预留

由于新开仓允许 AI 在可用资金 `40%-80%` 范围内决策，因此账本必须同时跟踪：

- 已实际占用保证金
- 尚未成交但已预留的虚拟保证金

未成交订单被撤销后，应释放相应预留。

## 7. 风险控制入口

V1 中风险控制应主要集中在以下几类开关：

- 全局交易开关
- 频道级自动执行开关
- 新开仓人工确认阈值
- 动作白名单
- 模拟盘 / 实盘切换确认
- 默认杠杆与资金投入比例范围

## 8. 对账与补偿

### 8.1 触发场景

- 系统重启后恢复
- WebSocket 中断后补偿
- 发现本地状态与交易所状态冲突

### 8.2 对账目标

- 真实订单状态
- 真实成交总量
- 真实持仓状态
- 虚拟持仓数量与事件链完整性

### 8.3 处理策略

- 先以交易所事实修正 `exchange_*`
- 再以 `exchange_*` 反推账本事件和虚拟状态
- 对无法自动修正的情况保留异常记录并提醒人工关注

## 9. V1 风险边界

V1 不做复杂策略风控引擎，重点是把以下边界守住：

- 不跨频道合并仓位逻辑
- 不在低置信度新开仓时自动放行
- 不让真实状态与虚拟状态长期分叉无告警
