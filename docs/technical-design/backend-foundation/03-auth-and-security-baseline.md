# 03. 认证与安全基线

## 1. 目标

本文件定义 Phase 2 必须落地的认证与安全最小基线，确保“单用户系统”仍具备可接受的本地安全能力和审计能力。

## 2. 认证模型

V1 认证模型固定为单用户密码认证：

- 登录口令为 `6` 位数字。
- 后端只存储强哈希结果，不存储明文或可逆密文。
- 登录成功后签发服务端会话。
- 前端通过会话接口获取认证事实，不自行保存敏感 token。

## 3. 密码存储基线

密码存储要求：

- 推荐算法：`Argon2id`。
- 必须启用随机盐。
- 必须设置合理成本参数（内存、迭代、并行度）。
- 禁止在日志中输出用户输入密码或哈希明文串。

密码变更要求：

- 变更前必须要求当前会话再验证或等效安全确认。
- 变更动作必须写入审计日志。

## 4. 会话管理基线

会话策略建议：

- 会话凭据通过 `HttpOnly` cookie 下发。
- Cookie 建议开启 `Secure`、`SameSite=Lax`（生产环境）。
- 会话必须有失效时间（绝对过期或滑动过期）。
- 登出必须使当前会话失效。
- 支持服务端主动失效（用于安全事件处理）。

会话对象建议字段：

- `session_id`
- `issued_at`
- `expires_at`
- `last_seen_at`
- `ip_hash`
- `user_agent_hash`

## 5. 登录限流与失败锁定

Phase 2 必须实现：

- 登录请求限流。
- 登录失败计数。
- 达阈值后临时锁定。

建议策略：

- 限流维度：`IP + endpoint`。
- 失败计数维度：单用户账户。
- 连续失败阈值：可配置，默认建议 `5`。
- 锁定时长：可配置，默认建议 `15` 分钟。

行为要求：

- 锁定期间返回明确错误码，例如 `AUTH_ACCOUNT_LOCKED`。
- 登录成功后重置失败计数。
- 锁定解除可依靠时间到期或管理员动作。

## 6. 审计与追溯

以下动作必须写入 `operator_actions` 或等效审计表：

- 登录成功与失败
- 登出
- 会话失效
- 设置变更
- 高风险开关变更

审计字段最低要求：

- `action`
- `target_type`
- `target_id`
- `result`
- `operator_source`
- `request_id`
- `correlation_id`
- `created_at`

## 7. 安全响应与错误暴露策略

错误暴露规则：

- 对认证失败返回统一语义，不泄露内部判定细节。
- 对锁定错误可返回剩余锁定秒数。
- 对内部异常返回通用错误码，详细堆栈只进服务端日志。

日志脱敏规则：

- 密钥、密码、会话凭据、签名串一律脱敏。
- 请求头仅保留必要字段并做白名单输出。

## 8. CSRF、CORS 与接口暴露边界

V1 推荐基线：

- 明确允许的前端来源（CORS 白名单）。
- 会话型写接口建议启用 CSRF 防护或等效机制。
- 禁止对公网暴露未鉴权调试接口。
- `/health/live` 可公开，`/health/deps` 建议受限访问。

## 9. 机密配置管理

机密配置（密码哈希参数、会话签名密钥、数据库密码等）要求：

- 仅从环境变量或密钥管理系统加载。
- 禁止写入前端可见接口。
- 禁止以明文写入版本库。

配置变更要求：

- 机密变更需记录变更来源和时间。
- 生产环境建议双人复核或等效流程。

## 10. Phase 2 安全验收清单

AI 侧验收建议：

- 密码强哈希存储检查通过。
- 多次失败触发锁定检查通过。
- 会话过期返回未认证状态检查通过。

人工侧验收建议：

- 正确密码可登录，错误密码不可登录。
- 锁定后在有效期内不可登录，解锁后可恢复。
- 退出后受保护接口不可访问。
- 日志中未出现明文敏感信息。
