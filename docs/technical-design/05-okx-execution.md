# 05. OKX 执行与同步设计

## 1. 设计目标

OKX 模块负责把结构化动作转成交易所可执行动作，并持续同步交易所真实状态。V1 重点是模拟盘先跑通、真实状态可信、订单映射清楚。

## 2. 模块拆分

建议拆为以下组件：

- `instrument_service`：标的查询与缓存
- `execution_planner`：动作拆分、订单计划生成
- `rest_gateway`：下单 / 撤单 / 查单 / 查余额 / 查持仓
- `ws_sync`：私有 WebSocket 状态同步
- `order_timeout_watcher`：超时撤单
- `execution_reconciler`：状态补偿对齐

## 3. 执行输入与输出

### 3.1 输入

- AI 决策结果
- 当前风险配置
- 当前环境配置
- 当前虚拟子仓位状态
- 当前真实订单和持仓状态

### 3.2 输出

- 一组订单计划
- 提交后的真实订单记录
- 后续同步得到的真实状态更新

## 4. 执行规划原则

### 4.1 限价优先

- 默认用限价单
- 仅在频道明确信号要求市价时使用市价单

### 4.2 一个动作可拆成多笔订单

例如以下情况需要拆单：

- 进场区间挂单
- 多个止盈点
- 分批止盈
- 附带止损止盈

### 4.3 先计划再提交

执行规划与实际提交应分层：

- 规划层只负责生成“应该怎么下”
- 网关层只负责“按计划提交到 OKX”

## 5. 订单模型映射

V1 需要支持的核心映射包括：

- 新开仓 -> 开仓主单
- 补仓 -> 追加主单
- 部分止盈 -> 减仓单 / 分批单
- 全部平仓 -> reduce-only 平仓单
- 止损 -> 止损条件单或等效模型
- 多止盈 -> 多笔止盈单
- 保本移动 -> 撤旧止损 + 新止损
- 改单 -> 撤旧单 + 新挂单或 OKX 原生改单

## 6. 客户端订单标识设计

每个真实订单请求都必须带 `cl_ord_id`，建议具备以下信息：

- 环境前缀
- 频道或虚拟持仓引用
- 时间片
- 顺序号或短随机串

要求：

- 同一真实请求唯一
- 能回溯到 `decision_id`
- 能支持幂等保护和异常对账

## 7. 状态同步

### 7.1 WebSocket 为主

私有 WebSocket 应作为订单、成交、持仓状态的主同步通道。

### 7.2 REST 为补偿

当 WebSocket 重连、丢包或启动恢复时，应使用 REST 做补偿查询。

### 7.3 同步目标

至少同步以下对象：

- 订单状态
- 成交明细
- 持仓状态
- 账户余额

## 8. 环境隔离

- 模拟盘和实盘分开配置密钥
- 订单、成交、快照都要带环境标识
- 控制台必须明确显示当前环境
- 从模拟盘切到实盘属于高风险动作

## 9. 超时撤单

V1 默认未成交订单 `5h` 自动撤销。建议实现为后台任务：

- 定期扫描仍处于 live 状态且超时的订单
- 发起撤单
- 写回状态和日志

## 10. 异常处理

### 10.1 可重试

- 查询类接口瞬时失败
- WebSocket 临时断连

### 10.2 不可盲重试

- 已提交但结果未知的下单请求
- 已发出撤单但状态未同步完成的订单

对于不可盲重试的场景，应先查单再决定下一步。
