# 05. Web 前端实现记录

## 1. 目的

本文件用于记录已经落地到代码仓库中的前端实现约定，避免后续只看到代码而不知道当时为什么这么拆。

它与前四份文档的关系如下：

- `01` 负责讲信息架构
- `02` 负责讲页面规格
- `03` 负责讲视觉系统
- `04` 负责讲 API 契约
- 本文件负责记录“当前代码已经怎么落了”

## 2. 当前代码位置

当前 Web 前端工作区位于：

- `apps/web`

这意味着后续后端、worker、脚本等其他工程可以继续在项目根目录下按并列目录扩展，而不需要把前端和后端硬塞在同一个包里。

## 3. 当前技术选型落地情况

已落地的前端基础选型：

- `React`
- `Vite`
- `TypeScript`
- `react-router`

当前阶段暂未接入真实后端，因此以下能力保留在下一阶段：

- 查询缓存层
- SSE 实时订阅层
- 表单校验和提交抽象层

## 4. 当前实现边界

首轮代码实现目标是“做出可继续扩展的正式壳层”，而不是“只做一组静态图”。

因此当前代码应至少满足：

- 有独立可运行的前端工作区
- 有正式路由结构
- 有登录页和受保护页面壳层
- 有 mock 数据驱动的核心页面
- 有统一设计 token

## 5. 当前落地约定

### 5.1 认证

当前登录是原型级实现：

- 接受任意 `6` 位数字密码
- 使用本地会话状态模拟登录
- 只用于让页面与路由工作流先跑起来

后续接真实后端时，必须替换为服务端会话方案。

### 5.2 数据来源

当前页面统一从 `apps/web/src/mock-data.ts` 读取数据。

这样做的目的是：

- 让页面先围绕稳定对象结构开发
- 让 mock 与真实 API 的替换边界清晰
- 避免每个页面各自维护一套临时假数据

### 5.3 视觉 token

当前设计 token 统一维护在：

- `apps/web/src/styles.css`

后续如果拆成更系统化的 token 文件，也应保证来源单一，避免页面私自定义颜色和间距。

### 5.4 路由

当前首批路由已按设计文档落地：

- `/login`
- `/overview`
- `/channels`
- `/logs`
- `/manual-confirmations`
- `/orders`
- `/virtual-positions`
- `/settings`

## 6. 2026-03-22 风格调整记录

本轮新增的明确设计决定如下：

- 整体风格改为“更可靠、更专业、更像控制台”
- 桌面端骨架固定为“顶栏 + 侧边栏 + 主工作区”
- 全局圆角整体收敛，不再保留强烈的大圆角语言
- 降低玻璃拟态、柔光和大面积装饰渐变
- 强化分区线、表格、列表和稳定栅格
- 首页从“带展示感的仪表盘”进一步收成“状态台 + 任务入口”

## 7. 对当前代码的影响说明

当前代码已经具备以下可复用基础：

- 路由结构
- 页面拆分
- mock 数据模型
- 认证原型

但视觉层与布局层需要在下一轮继续向新方向收口。重点调整范围如下：

- `apps/web/src/styles.css`
- `apps/web/src/components/AppShell.tsx`
- `apps/web/src/pages/OverviewPage.tsx`
- 必要时同步调整 `ChannelsPage`、`LogsPage`、`ManualConfirmationsPage`

也就是说，下一轮不是推翻前端工程，而是在现有骨架上做“样式语言与布局表达”的重构。

## 8. 下一阶段建议

当这版文档确认后，下一阶段建议按下面顺序继续：

1. 先把顶栏、侧边栏和主工作区的结构气质收紧到新风格。
2. 优先重做登录页与总览页，让新气质先成立。
3. 再统一修正各页面的圆角、边框、阴影、状态标签和按钮语言。
4. 最后再进入真实接口接入。

## 9. 维护原则

后续新增页面或改动现有页面时，建议遵守以下规则：

- 先改文档，再改结构性代码
- 首批字段尽量沿用 `04-api-contract-draft.md`
- 任何会影响真实/虚拟状态边界的界面调整，都应同步更新设计文档
- 任何会影响整体风格的调整，都优先收口到 `styles.css` 和壳层组件，而不是页面各自发散
