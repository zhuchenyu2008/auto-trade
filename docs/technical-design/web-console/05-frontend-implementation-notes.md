# 05. Web 前端实现记录

## 1. 目的

本文件用于记录已经落地到代码仓库中的前端实现约定，避免后续只看到代码而不知道当时为什么这么拆。

它与前四份文档的关系如下：

- `01` 负责讲信息架构
- `02` 负责讲页面规格
- `03` 负责讲视觉系统
- `04` 负责讲 API 契约
- 本文件负责记录“当前代码已经怎么落了”

## 2. 当前代码位置

当前 Web 前端工作区位于：

- `apps/web`

这意味着后续后端、worker、脚本等其他工程可以继续在项目根目录下按并列目录扩展，而不需要把前端和后端硬塞在同一个包里。

## 3. 当前技术选型落地情况

已落地的前端基础选型：

- `React`
- `Vite`
- `TypeScript`
- `react-router`

当前阶段暂未接入真实后端，因此以下能力保留在下一阶段：

- 查询缓存层
- SSE 实时订阅层
- 表单校验和提交抽象层

## 4. 当前实现边界

首轮代码实现目标是“做出可继续扩展的正式壳层”，而不是“只做一组静态图”。

因此当前代码应至少满足：

- 有独立可运行的前端工作区
- 有正式路由结构
- 有登录页和受保护页面壳层
- 有 mock 数据驱动的核心页面
- 有统一设计 token

## 5. 当前落地约定

### 5.1 认证

当前登录是原型级实现：

- 接受任意 `6` 位数字密码
- 使用本地会话状态模拟登录
- 只用于让页面与路由工作流先跑起来

后续接真实后端时，必须替换为服务端会话方案。

### 5.2 数据来源

当前页面统一从 `apps/web/src/mock-data.ts` 读取数据。

这样做的目的是：

- 让页面先围绕稳定对象结构开发
- 让 mock 与真实 API 的替换边界清晰
- 避免每个页面各自维护一套临时假数据

### 5.3 视觉 token

当前设计 token 统一维护在：

- `apps/web/src/styles.css`

后续如果拆成更系统化的 token 文件，也应保证来源单一，避免页面私自定义颜色和间距。

### 5.4 路由

当前首批路由已按设计文档落地：

- `/login`
- `/overview`
- `/channels`
- `/logs`
- `/manual-confirmations`
- `/orders`
- `/virtual-positions`
- `/settings`

## 6. 下一阶段建议

当这版前端骨架确认后，下一阶段建议按下面顺序继续：

1. 接入真实认证接口和会话状态。
2. 接入总览、频道、日志、人工确认四类 P0 数据接口。
3. 接入 SSE 实时事件流。
4. 再接真实订单、虚拟持仓和设置保存接口。

## 7. 维护原则

后续新增页面或改动现有页面时，建议遵守以下规则：

- 先改文档，再改结构性代码
- 首批字段尽量沿用 `04-api-contract-draft.md`
- 任何会影响真实/虚拟状态边界的界面调整，都应同步更新设计文档
