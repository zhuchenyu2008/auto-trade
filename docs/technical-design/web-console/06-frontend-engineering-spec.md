# 06. Web 前端实施级技术规范

## 1. 文档定位

本文件是 Web 控制台在 V1 阶段的“实施级”前端技术规范，目标是把信息架构、页面规格、视觉系统、API 草案和当前代码实现拉通，形成可以直接指导开发、联调、测试和验收的单一执行基线。

本文件重点回答以下问题：

- 前端工程应如何组织，才能支撑后续真实接口接入且避免重构返工。
- 每个页面的状态、动作、边界和错误处理应如何落地。
- 前后端契约、实时推送、风险操作、审计和可观测性在前端层面应如何统一实现。

## 2. 全局约束与术语

### 2.1 已确认业务边界（前端必须显式体现）

- 单用户、个人自用系统。
- 环境仅有 `paper` 与 `live`，并且必须在所有关键视图稳定可见。
- 新开仓低置信度进入人工确认；已有持仓跟进动作默认不进确认队列。
- 真实状态与虚拟状态分层展示，不得混成一张“综合仓位表”。
- 高风险动作必须二次确认并保留审计。

### 2.2 前端术语口径

- 真实层：`orders` / `fills` / `real_positions` 等交易所事实。
- 解释层：`virtual_positions` / 频道级 PnL / 消息解释链路。
- 链路键：`correlation_id`，用于跨页面追踪同一处理链路。
- 高风险动作：切换实盘、全局交易开关、强平、手动开仓、核心风控阈值变更。

## 3. 前端工程架构规范

### 3.1 当前工程位置

- Web 工程根目录：`apps/web`
- 入口：`src/main.tsx`
- 路由：`src/App.tsx`
- 全局状态容器：`src/state/AppStateContext.tsx`
- 全局样式与 token：`src/styles.css`

### 3.2 模块分层

建议采用以下职责分层（现有代码继续向此收敛）：

- `pages/`：页面容器，负责页面级编排，不承载复杂数据变换逻辑。
- `components/`：跨页面复用组件（壳层、抽屉、确认框、状态标签、状态面板）。
- `state/`：全局状态与应用行为编排（V1 原型期）。
- `lib/`：纯函数工具（格式化、文案标签、契约解码器）。
- `types.ts`：领域对象与枚举类型声明。

### 3.3 路由与壳层架构

V1 路由结构冻结为：

- `/login`
- `/overview`
- `/channels`
- `/logs`
- `/manual-confirmations`
- `/orders`
- `/virtual-positions`
- `/settings`

路由规则：

- 非登录路由统一经过 `ProtectedLayout`。
- 未登录访问受保护路由必须重定向到 `/login`。
- 未匹配路由在已登录域显示错误页，在未登录域重定向 `/login`。

壳层规则：

- 顶栏固定，必须持续显示环境、交易开关、健康状态、待确认数量、最新告警。
- 侧栏作为一级导航，结构稳定，不按页面动态重排。
- 顶栏提供全局高风险动作快捷入口；设置页危险操作区提供同等能力，二者均必须走统一确认模态。

### 3.4 运行时配置开关

前端运行时配置统一来自 Vite 环境变量：

- `VITE_DATA_SOURCE`：`mock` 或 `api`。
- `VITE_API_BASE_URL`：REST/SSE 基础路径（默认 `/api/v1`）。
- `VITE_ENABLE_SSE`：是否启用 SSE 实时订阅。

默认策略：

- 未配置时使用 `mock` 数据源，保证离线演示可用。
- 切到 `api` 时自动启用接口同步与 SSE（若开关允许）。

## 4. 状态管理规范

### 4.1 状态分层定义

状态按四层管理：

- 会话态：认证状态、当前环境、全局交易开关、健康状态、待确认数量。
- 领域态：频道、日志、确认项、真实订单、真实持仓、虚拟持仓、运行设置。
- UI 态：抽屉开关、筛选条件、表单草稿、当前选择项、局部动效高亮。
- 推导态：待确认数量、健康状态、过滤后列表、聚合 PnL。

### 4.2 推导规则

- `pendingManualConfirmationCount` 必须由确认项 `pending` 数量推导，不允许手工维护双写。
- `healthStatus` 由告警级别和频道错误摘要推导：
  - 任一 `error` -> `down`
  - 无 `error` 且有 `warning` 或频道错误摘要 -> `degraded`
  - 否则 -> `healthy`

### 4.3 会话持久化规则

- V1 原型允许使用 `localStorage` 保持登录态。
- 接入真实后端后，前端不存储敏感 token，改用服务端会话 cookie。
- 前端应通过 `/api/v1/auth/session` 获取会话事实并定期校验。

### 4.4 目标状态管理演进

从原型态演进到真实联调时，建议引入：

- 服务端状态：`TanStack Query`
- 表单：`react-hook-form`
- 领域写操作：统一 mutation 层并强制错误码处理

推荐查询键约定：

- `["session"]`
- `["overview","summary"]`
- `["channels", filters]`
- `["logs", filters]`
- `["manual-confirmations", filters]`
- `["manual-confirmation", confirmationId]`
- `["orders", filters]`
- `["fills", filters]`
- `["real-positions"]`
- `["virtual-positions", filters]`
- `["runtime-settings"]`

## 5. 领域模型与字段口径规范

### 5.1 枚举口径

前端必须与接口契约保持同一套枚举，不允许页面私自发明新值：

- `environment`: `paper` | `live`
- `health_status`: `healthy` | `degraded` | `down`
- `channel_status`: `enabled` | `disabled`
- `manual_confirmation_status`: `pending` | `approved` | `rejected` | `expired` | `invalidated`
- `order_status`: `pending` | `partially_filled` | `filled` | `canceled` | `rejected`
- `position_side`: `long` | `short`

### 5.2 字段语义口径

- 时间：ISO 8601 UTC 字符串（展示层本地化格式化）。
- 价格/数量/金额：字符串传输，前端显式格式化，不做隐式浮点口径变更。
- 关联链路：日志、确认项、订单、虚拟持仓均应暴露 `correlation_id`。
- 环境字段：凡涉及交易事实或高风险动作对象，必须显式带环境标识。

### 5.3 真实层与解释层的 UI 隔离要求

- 真实层页面标题和文案应明确“交易所事实视图”。
- 虚拟层页面标题和文案应明确“解释层虚拟视图”。
- 同屏出现两层数据时必须分区并标注层级，不允许仅靠颜色区分。

## 6. 页面级实施规范

### 6.1 登录页 `/login`

- 输入：6 位数字密码。
- 错误处理：失败提示、失败计数、临时锁定提示。
- 状态：
  - 未登录：可访问。
  - 已登录：重定向 `/overview`。
  - 锁定中：禁用提交并显示剩余秒数。
- 安全提示必须可见：会话失效回跳、敏感凭据不前端展示、实盘切换二次确认。

### 6.2 总览页 `/overview`

- 首屏目标：在最短时间回答环境风险、系统健康、待处理事项。
- 必须展示：环境、交易开关、健康状态、待确认数量、频道摘要、最近异常、最近动作。
- 可跳转入口：`/channels`、`/logs`、`/manual-confirmations`。
- 视觉优先级：风险条 > 关键摘要 > 队列入口 > 历史信息。

### 6.3 频道页 `/channels`

- 目标：频道生命周期管理 + 抓取健康观察。
- 必须支持：查询、筛选、新增、编辑、启停、详情抽屉。
- 删除策略：V1 采用逻辑删除或停用语义，不做硬删除。
- 列表区分：抓取异常 vs 暂无新消息必须可一眼区分。

### 6.4 日志页 `/logs`

- 目标：成为单链路排查入口。
- 必须支持筛选：频道、级别、模块、环境、`correlation_id`。
- 实时行为：
  - 新日志追加。
  - 历史查看时不可强制滚底。
  - 新增日志可短时高亮但不打断阅读。
- 详情抽屉展示：事件摘要、模块、频道、环境、时间、关联编号。

### 6.5 人工确认页 `/manual-confirmations`

- 布局：左队列 + 右详情。
- 队列操作：状态筛选、选择项切换、失效项可见。
- 详情必须包含：原始消息、上下文摘要、AI 决策、关键价格参数、环境、关联编号。
- 执行前校验：
  - 状态必须是 `pending`
  - 项目必须 `executable`
  - 会话环境必须与确认项环境一致
- 通过/拒绝均需二次确认。

### 6.6 真实订单页 `/orders`

- 目标：展示交易所事实（订单、成交、真实持仓）。
- 交互：点选订单打开详情抽屉。
- 抽屉必须显示：订单事实、执行形态、环境、可追踪信息。

### 6.7 虚拟持仓页 `/virtual-positions`

- 目标：展示频道级解释层状态与映射关系。
- 必须展示：虚拟持仓列表、聚合 PnL、映射说明、生命周期抽屉。
- 抽屉必须可回看：映射到的真实订单与关联编号。

### 6.8 设置页 `/settings`

- 目标：运行参数管理与风险边界配置。
- 分组：运行状态（只读）、模型与思考等级、交易参数、危险操作区。
- 运行状态分组仅展示“当前环境 / 全局交易状态”，不提供直接切换控件。
- 高风险动作（切环境、全局交易开关）仅可在危险操作区（或顶栏快捷入口）触发，且必须二次确认。
- AI API Key / Base URL 属于部署级机密配置，不在该页面明文填写。
- 配置保存应明确反馈“已保存并生效”或“已保存待生效（若后端定义）”。

## 7. 共用交互组件协议

### 7.1 `ConfirmModal`

- 仅用于高风险确认和强打断行为。
- 必须传入：标题、描述、确认文案、风险 tone、确认/取消回调。
- 不用于普通详情查看。

### 7.2 `Drawer`

- 用于对象详情与上下文补充。
- 触发行为：点行或点详情按钮。
- 关闭行为：遮罩点击、关闭按钮。

### 7.3 `StatusTag`

- 必须使用统一语义色：
  - `neutral`：中性状态
  - `success`：成功/健康/已通过
  - `warning`：告警/待确认/处理中
  - `danger`：错误/拒绝/高风险
  - `info`：一般信息强调

### 7.4 `StatePanel`

- 用于空态、加载态、错误态统一表达。
- 空态必须提供下一步动作提示。
- 错误态必须提供重试入口或回退入口。

## 8. API 集成实施规范

### 8.1 认证与会话

- 登录：`POST /api/v1/auth/login`
- 会话读取：`GET /api/v1/auth/session`
- 登出：`POST /api/v1/auth/logout`

约束：

- 会话由服务端 cookie 驱动。
- 前端不在本地存储敏感令牌。
- 401/419 统一触发会话失效流程并回跳登录页。

### 8.2 响应包络处理

所有请求统一按以下结构解包：

- `data`: 业务数据
- `meta`: 请求元信息（`request_id`、`server_time`）
- `error`: 错误对象

处理规则：

- `error != null` 视为业务失败，即使 HTTP 状态为 200。
- `request_id` 应在前端错误日志中透传，便于后端排查。

### 8.3 查询与分页

列表查询统一支持：

- `cursor`
- `limit`
- 业务筛选参数（如 `from/to/level/module/channel_id/environment/correlation_id`）

前端要求：

- 支持“刷新当前游标页”与“回到第一页”。
- 筛选条件变更时重置游标。

### 8.4 写操作与幂等

以下写操作建议前端附带幂等键（例如请求头 `X-Idempotency-Key`）：

- 人工确认通过/拒绝
- 高风险开关切换
- 频道新增/更新

冲突处理：

- 若返回“已处理/已失效/状态冲突”错误码，前端必须刷新详情并提示当前状态，不得盲目重试。

写操作审计透传（建议）：

- `X-Operator-Source: web-console`
- `X-Idempotency-Key: <key>`
- `X-Correlation-Id: <corr_id>`
- `X-Audit-Action: <action_name>`
- `X-Audit-Target: <target_id>`

### 8.5 错误码前端处理基线

前端必须针对以下错误码建立明确处理：

- `MANUAL_CONFIRMATION_EXPIRED`
- `MANUAL_CONFIRMATION_INVALIDATED`
- `ENVIRONMENT_MISMATCH`
- `ACTION_NOT_ALLOWED`
- `SESSION_INVALID`
- `RATE_LIMITED`

## 9. SSE 实时推送实施规范

### 9.1 连接策略

- 主通道：`GET /api/v1/events/stream`
- 断线重连：指数退避，建议 `1s -> 2s -> 5s -> 10s -> 20s`，上限 `30s`
- 长时间断线时在顶栏显示“实时流已降级”提示

### 9.2 事件类型与前端动作映射

推荐映射：

- `system.snapshot_updated` -> 刷新 `session` / `overview` 摘要
- `log.appended` -> 追加日志列表或触发日志查询失效
- `channel.health_changed` -> 刷新频道摘要与频道列表
- `manual_confirmation.changed` -> 刷新确认队列与详情
- `order.changed` -> 刷新订单/成交/真实持仓
- `virtual_position.changed` -> 刷新虚拟持仓和聚合 PnL

### 9.3 顺序与一致性

- UI 以“最终一致”展示为目标，接收事件后优先局部更新并定期全量校准。
- 对同对象事件若出现乱序，按 `occurred_at` 与服务端状态回读结果兜底。

### 9.4 降级策略

- SSE 不可用时回退轮询。
- 轮询间隔按页面重要度分级：
  - 总览/确认/日志：5-10s
  - 频道/订单/持仓：10-20s
  - 设置：按需刷新

## 10. 安全、风险与审计前端规范

### 10.1 敏感信息处理

- 前端不展示明文密钥、令牌、密文密码、完整 cookie。
- 前端不通过运行时设置接口下发或回显 AI API Key / Base URL 等机密字段。
- 错误提示不得拼接后端原始异常堆栈中的敏感字段。

### 10.2 高风险动作保护

以下动作必须使用确认模态并有风险文案：

- 切换到实盘
- 开启全局交易
- 强平（后续页面接入时）
- 手动开仓（后续页面接入时）
- 核心阈值变更

### 10.3 环境一致性校验

- 人工确认执行前必须校验确认项环境与当前会话环境一致。
- 若不一致，按钮禁用并显示显式告警。

### 10.4 审计可追溯要求

前端触发关键动作时必须保证后端可记录：

- 动作入口（Web）
- 目标对象
- 操作结果
- `correlation_id` 或等效关联键
- 环境标识

## 11. 可观测性与错误处理规范

### 11.1 前端日志字段建议

前端自身日志（开发/错误采集）建议包含：

- `timestamp`
- `level`
- `module`（page/component/api/sse）
- `request_id`（若有）
- `correlation_id`（若有）
- `environment`
- `message`

### 11.2 统一错误处理

- 网络错误：提示“网络异常，可重试”。
- 会话错误：清会话并回登录。
- 业务错误：按错误码转为可执行文案。
- 未知错误：展示简化提示并保留 `request_id` 供排查。

### 11.3 页面三态一致性

所有页面必须可验证以下状态：

- 空状态：说明下一步操作。
- 加载状态：局部骨架，不整页闪烁。
- 错误状态：可重试，且标明失败对象。

## 12. 性能与可访问性规范

### 12.1 性能基线

- 首屏应优先保证总览核心信息先可见。
- 日志和大表格避免一次性渲染全部历史数据。
- 动效仅服务于层级与状态反馈，不做重装饰动画。

### 12.2 大列表处理建议

- 日志行数超过阈值（建议 500）时启用虚拟列表方案。
- 采用分页或 cursor，不允许无限全量加载。

### 12.3 响应式基线

- 桌面端保留完整壳层结构。
- 平板可折叠侧栏。
- 手机端保证总览查看和人工确认闭环，不追求完整并列表格体验。

### 12.4 可访问性基线

- 焦点可见。
- 关键按钮触控面积充足。

## 14. 当前实现差距（截至 2026-03-22）

以下为当前 `apps/web` 与目标规范差距，后续需按优先级修正：

1. 已接入可切换 API/SSE 基础层，但服务器状态仍集中在 `AppStateContext`，尚未迁移到 `TanStack Query`。
2. SSE 客户端与降级轮询已接入，但需与后端真实事件流完成联调校验。
