# 01. Web 控制台信息架构

## 1. 目标

本文件定义 Web 控制台的信息架构，回答三个问题：

1. 使用者每天进入系统后，第一眼应该看到什么。
2. 各个页面之间如何组织，才能覆盖主要工作路径。
3. 桌面端与移动端分别怎样保证信息可用性。

本设计默认系统只有一个实际使用者，因此信息架构不围绕角色切换，而围绕工作任务切换展开。

## 2. 使用者心智模型

单人使用场景下，控制台的核心任务可以归纳为三个连续动作：

1. 先确认风险边界是否安全。
2. 再判断系统是否正常运行。
3. 最后处理具体信号、订单和配置动作。

因此首层信息架构应围绕以下四类对象组织：

- 系统边界：环境、交易开关、系统健康、最近异常
- 信号来源：频道、抓取状态、最近消息处理结果
- 执行动作：人工确认、真实订单、真实成交
- 解释层状态：虚拟子仓位、频道级 PnL、消息解释链路

## 3. 全局布局

### 3.1 桌面端布局

桌面端采用“三段式工作台”布局：

- 顶部全局状态栏：始终固定显示最高优先级风险与系统状态
- 左侧主导航：在页面之间切换，不承担明细展示
- 中央主工作区：承担当前页面核心内容

对于日志详情、确认详情、订单详情等需要保留上下文的场景，优先使用右侧抽屉，而不是整页跳转或过多弹窗。

### 3.2 顶部全局状态栏

顶部状态栏固定显示以下内容：

- 当前环境：`模拟盘` 或 `实盘`
- 全局交易开关：`开启` / `关闭`
- 系统健康状态：`正常` / `降级` / `异常`
- 待人工确认数量
- 最近关键异常摘要

这条状态栏是整个控制台中优先级最高的区域，任何页面都不能把它隐藏掉。

### 3.3 左侧导航结构

左侧导航按“观察优先、操作次之、配置最后”的顺序组织：

- 总览
- 频道
- 日志
- 人工确认
- 真实订单
- 虚拟持仓
- 系统设置

导航中不单独设置“用户中心”“权限管理”“团队协作”等多用户后台常见入口。

## 4. 路由树

建议路由结构如下：

| 路由 | 页面 | 优先级 | 用途 |
| --- | --- | --- | --- |
| `/login` | 登录页 | P0 | 单用户登录入口 |
| `/overview` | 总览页 | P0 | 每日进入系统后的默认页 |
| `/channels` | 频道列表页 | P0 | 查看与维护频道配置 |
| `/logs` | 日志页 | P0 | 查看实时处理链路与异常 |
| `/manual-confirmations` | 人工确认页 | P0 | 处理新开仓低置信度确认 |
| `/orders` | 真实订单与成交页 | P1 | 查看交易所真实执行结果 |
| `/virtual-positions` | 虚拟持仓与 PnL 页 | P1 | 查看频道级解释层状态 |
| `/settings` | 系统设置页 | P1 | 查看和修改运行参数 |

V1 不建议在首轮就做大量多级子路由。能通过页面内 tab、抽屉或筛选完成的，不额外拆新页面。

## 5. 页面之间的跳转关系

主要工作路径建议如下：

1. 从总览页进入异常或待确认项。
2. 从待确认项进入确认详情后完成确认或拒绝。
3. 从日志页根据 `correlation_id` 下钻到订单、消息或确认详情。
4. 从频道页进入频道详情抽屉，查看最近抓取状态和最近处理记录。
5. 从真实订单页或虚拟持仓页回看关联消息和决策链路。

也就是说，控制台的信息流不应是孤立页面集合，而应围绕 `correlation_id` 形成可追踪链路。

## 6. 关键页面职责分工

### 6.1 总览页

总览页负责回答“现在系统是否正常、哪里需要我立刻介入”。

它不负责承载完整历史明细，而负责：

- 当前环境和全局开关状态
- 频道健康摘要
- 最近异常
- 待确认项摘要
- 最新决策和执行摘要

### 6.2 频道页

频道页负责回答“系统正在监听什么、各频道是否健康、配置是否正确”。

它不负责实时日志主视图，也不负责完整订单执行明细。

### 6.3 日志页

日志页负责回答“刚才发生了什么、失败在哪里、这一条链路是怎么走的”。

它是排查问题的第一入口，因此必须支持：

- 实时追加
- 级别筛选
- 模块筛选
- 频道筛选
- `correlation_id` 检索

### 6.4 人工确认页

人工确认页负责回答“有哪些新开仓动作在等我决定、每个待确认项的上下文是否足够”。

它不应该混入普通日志流，也不应该要求频繁切页才能完成操作。

### 6.5 真实订单页

真实订单页负责查看交易所层面的真实事实，包括：

- 真实订单状态
- 真实成交结果
- 当前真实持仓

### 6.6 虚拟持仓页

虚拟持仓页负责查看系统解释层状态，包括：

- 频道虚拟子仓位
- 频道级 PnL
- 频道动作与真实订单的映射

### 6.7 设置页

设置页负责管理系统运行参数，不承担实时监控职责。

## 7. 明细展示方式

V1 建议优先采用以下展示方式：

- 列表页承载“筛选、排序、浏览”
- 抽屉承载“对象详情与上下文补充”
- 模态框只用于高风险确认和强打断动作

以下场景优先使用抽屉：

- 查看确认项详情
- 查看日志链路详情
- 查看某频道最近抓取状态
- 查看某虚拟持仓生命周期

以下场景优先使用模态框：

- 切换到实盘
- 强平
- 手动开仓
- 修改高影响阈值

## 8. 响应式策略

### 8.1 桌面端优先

V1 首轮设计以桌面端为主，因为主要操作包括：

- 高频查看日志
- 处理人工确认
- 对比真实与虚拟状态
- 维护配置

这些任务都需要较高的信息密度。

### 8.2 移动端策略

移动端需要保证“可看、可确认、可做低风险操作”，但不强求完整替代桌面端体验。

移动端最低要求包括：

- 查看总览摘要
- 查看待确认项数量和详情
- 处理确认或拒绝
- 查看频道和最近异常

移动端首轮不要求提供舒适的高密度日志排查体验，也不要求复杂的多列表并排布局。

## 9. 空状态、错误状态和加载状态

V1 各页面必须统一处理以下三类状态：

- 空状态：例如尚未配置频道、暂无待确认项、暂无订单记录
- 错误状态：例如日志流断开、频道列表加载失败、确认详情失效
- 加载状态：例如页面首次加载、抽屉详情加载、筛选切换中的局部刷新

状态处理原则如下：

- 空状态要告诉使用者“下一步该做什么”
- 错误状态要说明失败对象和可重试方式
- 加载状态优先使用局部 skeleton，不整页闪烁

## 10. 信息架构验收要点

当首版原型完成后，至少应满足以下检查项：

- 从总览页能在三次点击内进入任何待确认项详情
- 从日志页能通过 `correlation_id` 找到对应决策或订单链路
- 从虚拟持仓页能明确知道其对应的是解释层而非交易所真实状态
- 高风险动作不会与普通操作按钮混在同一视觉层级
- 在未配置频道、无待确认项、无订单记录时，页面仍然可理解
