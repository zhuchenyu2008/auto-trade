# 07. Web 控制台设计

## 1. 文档定位

本章是 Web 控制台相关技术设计的父文档，负责说明前端在 V1 中的定位、总体原则、实施顺序和子文档地图。

由于本项目明确采用“前端优先”的推进顺序，本章不追求一次性写完所有页面细节，而是先把最容易影响后续返工成本的内容定下来：

- 信息架构
- 视觉与交互方向
- 页面级职责边界
- 前后端契约草案

具体页面与组件细节不再继续堆在本文件中，而是在 `web-console/` 子目录分拆维护。

## 2. 设计目标

Web 控制台是 V1 的主操作入口，目标不是做一个通用 SaaS 管理后台，而是做一个供单人长期使用的交易操作台。其设计必须同时满足以下目标：

- 让使用者在一个界面里完成观察、确认、配置和日常运维
- 让“真实交易所状态”和“频道虚拟状态”始终清楚分层
- 让高风险动作在视觉上足够醒目，避免误操作
- 让首批页面可以在后端未完全就绪前先用原型和假数据定稿
- 让后续接入真实数据时尽量不推翻首版界面结构

## 3. 总体技术路线

V1 Web 控制台推荐采用以下技术组合：

- 框架：`React + Vite`
- 路由：`react-router`
- 服务端状态：`TanStack Query`
- 表单：`react-hook-form`
- 实时更新：优先 `SSE`，保留 `WebSocket` 扩展位
- 设计系统：项目内自定义 token + 组件层，不直接依赖重型后台模板

前端首版应按“契约先行 + 假数据驱动”的方式推进，即先把页面结构、字段语义和状态枚举定下来，再逐步把页面从 mock 数据切换到真实 API。

## 4. 跨子文档通用原则

### 4.1 单用户工作台，而非多用户后台

- 不设计多用户、多角色、权限中心、协作审批等后台概念。
- 认证、会话和审计仍然保留，但它们是为了安全和可追溯，而不是为了组织协作。

### 4.2 信息密度优先，但层级必须明确

- 控制台应优先保证“扫一眼就知道现在系统在干什么”，而不是把大量卡片堆在首页。
- 关键状态优先级应固定为：环境状态、全局交易开关、异常、待确认项、频道健康、订单与持仓。

### 4.3 真实状态与虚拟状态分层展示

- 真实订单、真实成交、真实持仓始终代表交易所层面的事实。
- 频道虚拟子仓位、频道级 PnL、消息解释结果始终代表系统内部解释层。
- 前端不得使用同一种视觉结构把这两层数据混成一个“综合仓位表”。

### 4.4 高风险动作必须强提示

- 环境切换到实盘
- 手动强平
- 手动开仓
- 修改关键风控阈值

以上动作必须在视觉上显著区别于普通按钮，并统一进入强确认流程。

### 4.5 前端优先并不意味着前端独立拍板

- 页面结构可以先行
- 字段名和状态枚举需要尽早稳定
- 一旦进入真实数据接入阶段，应优先遵守已冻结的页面契约，减少前后端反复返工

## 6. 实施顺序

Web 控制台建议按以下顺序推进：

1. 先完成视觉方向、应用骨架和信息架构确认。
2. 再完成登录页、总览页、频道管理页、日志页、人工确认页的高保真原型。
3. 然后冻结首批 API 契约，接入认证、频道、日志、人工确认和系统总览数据。
4. 最后补真实订单、虚拟持仓、PnL、系统设置等后续页面。

这个顺序与需求文档第十一章保持一致，核心目的不是“先做看起来漂亮的东西”，而是先把以后每天要面对的工作界面打磨对。

## 7. 与后端的主要边界

Web 控制台在 V1 首批依赖的后端能力主要有：

- 登录与会话校验
- 系统总览摘要
- 频道列表与频道配置维护
- 日志查询与实时事件推送
- 人工确认队列与确认详情
- 全局开关、环境状态和系统健康状态

第二批再接入：

- 真实订单与真实成交查询
- 频道虚拟子仓位与 PnL 查询
- 运行参数与风控配置维护

## 8. 非目标

以下内容不属于 V1 Web 控制台首轮设计目标：

- 多用户协作 UI
- 角色权限配置 UI
- 面向外部宣传的官网型页面
- 大而全的指标分析中心
- 高自由度拖拽布局
- 复杂图表工作台

V1 的重点是把“稳定使用、快速判断、避免误操作”做到位，而不是把控制台做成一套可无限配置的通用平台。
