# 08. OpenClaw 与运维协作设计

## 1. 目标

OpenClaw 在 V1 中不是主操作面，而是一个轻量级协作 / 远程入口。目标包括：

- 在 Telegram topic 中输出关键日志摘要
- 提供自然语言状态查询
- 提供少量低风险控制动作
- 对高风险动作执行确认或直接拒绝

## 2. 能力边界

### 2.1 可直接支持的能力

- 查询系统状态
- 查询频道状态
- 查询最近日志
- 查询当前持仓、挂单、PnL
- 查询最近消息解释
- 暂停频道
- 恢复频道

### 2.2 必须确认或禁止直执的能力

- 切换实盘
- 强平
- 手动开仓
- 修改核心风控参数
- 其他会直接改变真实资金风险边界的动作

## 3. Topic 设计建议

V1 可先按两个逻辑 topic 组织：

- 日志 topic：自动输出摘要
- 运维 topic：查询与控制

即使物理上首版先只实现一个 topic，也建议在逻辑上先区分输出类型。

## 4. Skill 设计建议

项目 skill 建议至少覆盖以下能力：

- 查最近系统日志
- 查某频道最近处理结果
- 查某标的的真实持仓与虚拟持仓
- 查某条消息对应的 AI 决策摘要
- 暂停 / 恢复频道
- 引导高风险动作进入确认路径

## 5. 后端集成方式

OpenClaw 层不应直接操作数据库，而应调用后端 API 或内部应用服务。

建议模式：

- 查询类 -> 调用只读查询服务
- 控制类 -> 调用应用服务并写审计记录
- 高风险类 -> 返回“需要确认”或触发确认流程

## 6. 审计要求

所有通过 OpenClaw 发起的动作都应记录：

- 触发时间
- 入口：OpenClaw
- 动作类型
- 目标对象
- 最终结果
- `correlation_id`

## 7. 输出安全要求

- 不返回敏感密钥
- 不返回完整凭据
- 异常信息应脱敏
- 对不允许的动作给出明确原因，而不是静默失败

## 8. 与 Web 控制台的一致性要求

- 高风险动作边界一致
- 当前环境显示一致
- 配置开关状态一致
- 审计记录口径一致

OpenClaw 不应形成第二套独立规则，否则后续很难维护。
