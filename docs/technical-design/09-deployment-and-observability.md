# 09. 部署、配置与可观测性设计

## 1. V1 部署目标

V1 的部署目标不是做大规模高可用集群，而是：

- 稳定运行
- 容易部署
- 容易恢复
- 容易排障

## 2. 推荐部署拓扑

建议的最小部署单元：

- 前端静态资源 / 前端服务
- FastAPI 主服务
- 后台任务进程
- OKX 同步进程
- PostgreSQL
- Redis

开发环境可合并，正式长期运行建议至少把数据库和 Redis 独立出来。

## 3. 配置分组

### 3.1 机密配置

- OKX 模拟盘凭据
- OKX 实盘凭据
- 模型服务密钥
- Telegram 受保护接入凭据（若未来扩展）

### 3.2 业务配置

- 频道配置
- 模型与思考等级
- 默认杠杆
- 确认阈值
- 上下文窗口长度
- 全局交易开关

### 3.3 系统状态

- 抓取 cursor
- 最近错误信息
- 快照状态
- 队列状态

## 4. 可观测性设计

### 4.1 日志

日志至少应分为：

- 应用日志
- 业务事件日志
- 审计日志

建议每条关键日志带：

- 时间戳
- 级别
- 模块名
- `correlation_id`
- 环境标识

### 4.2 健康检查

建议提供：

- `/health/live`：进程是否活着
- `/health/ready`：数据库、Redis、关键依赖是否可用
- `/health/deps`：外部依赖摘要状态

### 4.3 实时状态

- Web 控制台通过 `SSE` 或等效方案接收日志和关键状态流
- OpenClaw 接收摘要级运行状态

## 5. 恢复策略

### 5.1 服务重启恢复

重启后至少应恢复：

- 频道抓取 cursor
- 当前环境配置
- 未完成订单状态
- 当前真实持仓
- 人工确认队列
- 虚拟子仓位状态

### 5.2 数据备份建议

V1 至少应保证 PostgreSQL 可定期备份。重点保护对象包括：

- 消息版本
- AI 决策
- 订单和成交
- 账本事件
- 审计日志

## 6. 监控重点

V1 最值得盯的指标不是 CPU，而是链路健康度，例如：

- 频道抓取成功率
- AI 调用失败率
- 订单提交失败率
- WebSocket 重连频率
- 待确认项堆积数量
- 账本对账异常数量

## 7. 发布策略建议

- 先在模拟盘环境完成完整回归
- 再进入实盘准备，而不是直接实盘联调
- 高风险配置变更应单独演练
- 文档、配置和数据库迁移应一起版本化

## 8. 后续可扩展项

后续若系统进入长期运行阶段，可再补：

- 指标看板
- 自动告警
- 定时备份校验
- 更细粒度的错误分级
- 更完整的发布回滚策略
