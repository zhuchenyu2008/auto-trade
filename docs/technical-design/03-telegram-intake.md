# 03. Telegram 接入与消息处理设计

## 1. 范围

V1 仅实现 Telegram 公开频道网页抓取，不实现 Bot API 或 TDLib。该模块目标是稳定发现新消息与编辑消息，并把页面信息转成后续可处理的数据。

## 2. 组件拆分

建议拆为以下子组件：

- `channel_scheduler`：按频道调度抓取任务
- `page_fetcher`：抓取 `t.me/s/<channel>` 页面
- `message_parser`：从 HTML 中提取消息块、消息标识、正文和可见时间
- `cursor_manager`：管理每个来源的 last seen 状态
- `change_detector`：识别新消息、编辑消息和无变化抓取
- `normalization_dispatcher`：把有效变化交给标准化模块

## 3. 轮询策略

### 3.1 基本策略

- 默认轮询周期 `30s`
- 频道级独立调度
- 每轮抓取记录开始时间、结束时间、成功 / 失败结果

### 3.2 调度约束

- 同一频道同一时刻只允许一个抓取任务在跑
- 抓取失败不阻塞其他频道
- 可在频道级增加轻微随机抖动，避免所有频道同一秒打到页面

## 4. 页面解析策略

V1 不追求把页面所有装饰信息都解析出来，优先抓这几类字段：

- 来源消息标识
- 文本正文
- 编辑后正文
- 可见时间信息
- 页面中可识别的附件 / 链接摘要（若后续需要）

解析结果优先保证稳定，其次再追求字段丰富度。

## 5. 新消息与编辑消息判定

### 5.1 新消息

当页面中出现某个 `source_message_id`，且本地不存在该消息时，判定为新消息。

### 5.2 编辑消息

当本地已存在该 `source_message_id`，但正文内容或关键可比较内容发生变化时，判定为编辑消息。

### 5.3 无变化重复抓取

当本地已存在该消息，且比较后无差异时，判定为重复无变化，不进入后续 AI 链路。

## 6. cursor 设计

`source_cursors` 主要负责两个目标：

- 限制“从哪里开始看新内容”
- 帮助系统重启后快速恢复

V1 不必过度迷信单一 cursor 数值，可采用“最后已确认处理的消息范围 + 最近抓取状态”的组合方式。

## 7. 错误处理

### 7.1 可重试错误

- 网络超时
- 临时 DNS / 连接异常
- 页面瞬时加载失败

### 7.2 需记录并人工关注的错误

- 页面结构变化导致解析失败
- 长时间抓不到新内容但页面可访问
- 消息块字段异常缺失

## 8. 输出契约

抓取模块输出到标准化模块的最小对象应包括：

- `channel_id`
- `source_type`
- `source_message_id`
- `raw_content`
- `fetched_at`
- `detected_change_type`
- `raw_payload_ref` 或等效原始快照引用

## 9. 后续扩展位

尽管 V1 不做官方接入，但实现时仍应保留来源适配层接口，例如：

- `fetch_messages()`
- `parse_messages()`
- `build_cursor()`

这样后续接入 Bot API 或 TDLib 时，无需推翻整个消息处理管道。
