# 自动交易系统需求文档（V1 草稿）

## 1. 文档概述

### 1.1 编写目的

本文档用于定义自动交易系统 V1 的业务范围、核心规则、主要功能、非功能约束和验收口径。本文档的定位是产品需求文档草稿，服务于以下工作：

- 为产品、开发、测试和运维提供统一的目标和边界说明
- 为后续技术方案设计、数据库设计、接口设计和任务拆分提供输入
- 为 V1 范围控制和阶段性交付提供判断依据
- 为上线前验收提供统一的检查标准

本文档当前为第一版草稿，优先明确“系统要做什么、做到什么程度、哪些暂时不做”，避免在实现早期出现范围漂移。

### 1.2 项目背景

本项目的目标是搭建一套围绕 Telegram 频道交易信号的自动化交易系统。系统从公开频道中抓取消息，基于同频道上下文调用 AI 进行结构化理解，再将解析得到的交易意图映射为 OKX 合约交易动作，并通过 Web 控制台和 OpenClaw 提供个人运维、人工确认、查看和受控管理能力。

从业务上看，该系统并不是一个通用量化平台，而是一个“频道信号跟随与管理系统”。系统的核心价值不在于生成交易观点，而在于：

- 稳定地获取频道消息并追踪其变化
- 尽量准确地把自然语言信号还原成结构化交易意图
- 在单账户约束下，管理多个频道同时存在的虚拟持仓关系
- 把复杂的交易跟进动作持续映射为 OKX 可执行动作
- 在自动执行和人工干预之间维持清晰边界和可审计性

### 1.3 V1 建设目标

V1 的目标不是一次性覆盖所有可能的接入方式、交易所和场景，而是先形成一条可运行、可观察、可审计、可扩展的最小闭环。V1 应至少达成以下目标：

1. 能稳定监控一个或多个公开 Telegram 频道，并识别新消息与编辑消息。
2. 能在单频道上下文中调用 AI，对消息进行结构化解析并给出明确动作建议。
3. 能在 OKX 上执行合约相关的开仓、补仓、减仓、平仓、止损、止盈和撤改单等动作。
4. 能在单账户、单向持仓模式下，用“频道虚拟子仓位账本”解释和管理不同频道的交易生命周期。
5. 能通过 Web 控制台完成基本配置、人工确认、查看、切换和个人运维操作。
6. 能通过 OpenClaw 在 Telegram topic 中提供日志输出、状态查询和有限控制能力。

### 1.4 适用范围

本文档仅覆盖 V1 范围。V1 的目标是先交付一个能跑通完整链路、适合后续扩展的基础版本，因此在接入面和交易范围上采取明确收敛策略。

V1 覆盖范围包括：

- Telegram 公开频道网页抓取
- 同频道上下文构建与 AI 结构化决策
- OKX 合约 / 永续合约交易执行
- 单账户、单向持仓、逐仓模式下的真实仓位管理
- 频道级虚拟子仓位账本
- 单用户 Web 控制台
- OpenClaw 日志、查询与受控操作
- 日志、审计、恢复与基础安全控制

V1 不覆盖范围包括：

- Telegram Bot API 正式接入
- TDLib / 用户会话接入
- 多交易所接入
- 多账户管理
- 现货交易
- 双向持仓模式
- 多用户、多角色和权限体系
- 跨频道优先级和跨频道合并决策

### 1.5 文档使用对象

本文档主要面向以下读者：

- 产品负责人：确认需求范围、优先级和阶段目标
- 开发人员：理解业务约束、功能边界和模块职责
- 测试人员：编写验收用例和回归范围
- 后续协作方：理解项目的目标、约束和演进方向

### 1.6 术语说明

为避免后续描述歧义，V1 中主要术语定义如下：

- 频道：指被系统监控的单个 Telegram 公开频道，是消息抓取、上下文构建、决策判断和虚拟子仓位归属的基本隔离单位。
- 消息版本：指同一条频道消息在不同时间点的内容版本。消息发生编辑后，应保留新旧版本关系。
- AI 决策：指系统将单条消息及其同频道上下文发送给模型后，得到的结构化结果，包括动作类型、价格信息、风险信息、置信度和理由等。
- 新开仓：指在某频道的虚拟子仓位上下文中，被判定为首次建立新的交易方向敞口的动作。
- 跟进动作：指针对已有频道虚拟子仓位的后续管理动作，如补仓、减仓、平仓、移动止损、设置止盈、撤单、改单等。
- 虚拟子仓位：指系统内部按频道维护的持仓抽象层。交易所真实账户只存在净仓位，但系统内部需按频道记录各自的持仓生命周期和盈亏状态。
- 真实净仓位：指 OKX 单账户在单向持仓模式下的真实持仓结果，是交易所层面的最终状态。
- 人工确认：指系统对部分交易决策不直接执行，而进入人工确认流程后再决定是否执行。V1 中仅针对新开仓低置信度信号启用。
- 模拟盘 / 实盘：指 OKX 的两套运行环境。二者在凭据、配置、风险和审计要求上必须隔离。
- 高风险动作：指可能直接影响真实资金风险暴露或改变核心运行边界的动作，例如切换到实盘、强制平仓、手动开仓、修改核心风控策略等。

## 2. 产品范围与边界

### 2.1 产品定位

本系统的 V1 产品定位是“面向 Telegram 交易信号的个人自动化执行与管理平台”。系统不承担独立生成投资策略的职责，而是承担以下职责：

- 接收外部频道提供的交易信号
- 将非结构化自然语言转为结构化交易意图
- 在用户确认的规则边界内自动执行或辅助执行
- 将执行、跟进、审计和观察统一纳入一个闭环系统

因此，V1 的重点不是“策略研究”，而是“信号接入、结构化理解、执行映射、状态管理和个人运维控制”。

### 2.2 In Scope

V1 必须覆盖以下范围：

#### 2.2.1 Telegram 网页抓取接入

- 支持对一个或多个公开频道进行网页抓取
- 支持维护抓取游标和抓取状态
- 支持识别新消息和编辑消息
- 支持保存原始消息、标准化消息和消息版本

#### 2.2.2 消息标准化与上下文构建

- 对不同频道抓取结果统一成标准消息结构
- 在单频道范围内构建最近相关上下文
- 注入该频道当前未完成信号和虚拟子仓位状态
- 为 AI 生成稳定、可审计的输入快照

#### 2.2.3 AI 结构化决策

- 支持运行时切换模型
- 支持运行时切换思考等级
- 支持输出结构化 JSON 结果
- 覆盖开仓、补仓、减仓、平仓、止损、止盈、撤改单等动作
- 对新开仓低置信度场景给出人工确认判定

#### 2.2.4 OKX 执行与状态同步

- 支持 OKX 合约 / 永续合约交易
- 支持杠杆、下单、撤单、改单、止损止盈等执行能力
- 支持订单、成交、持仓和账户状态同步
- 支持模拟盘与实盘切换

#### 2.2.5 虚拟子仓位账本

- 支持按频道维护独立的持仓生命周期
- 支持记录虚拟均价、挂单映射、成交映射和盈亏估算
- 支持将多个频道动作映射到单账户真实净仓位
- 支持保留完整审计关系

#### 2.2.6 Web 控制台

- 支持单用户登录鉴权、会话管理和登录安全限制
- 支持频道管理、模型配置、环境切换和全局开关
- 支持日志、仓位、挂单、PnL 和人工确认查看及处理

#### 2.2.7 OpenClaw 集成

- 支持在 Telegram 群组 topic 中输出日志摘要
- 支持自然语言查询系统状态
- 支持有限的低风险控制动作
- 对高风险动作执行二次确认和审计记录

#### 2.2.8 安全、审计与恢复能力

- 支持关键事件的日志和关联 ID 记录
- 支持人工操作的审计追溯
- 支持系统重启后的关键状态恢复
- 支持日志脱敏和敏感配置隔离

### 2.3 Out of Scope

为保证 V1 能够聚焦交付，以下内容明确不在当前阶段范围内：

#### 2.3.1 非网页抓取类 Telegram 官方接入

V1 仅实现 Telegram 网页抓取。Telegram Bot API、TDLib 和用户会话方式仅保留架构扩展位，不在 V1 交付清单内。

#### 2.3.2 多交易所与多账户能力

V1 只支持 OKX 单账户。其他交易所接入、账户间隔离和多账户资金分配均不纳入当前范围。

#### 2.3.3 非合约场景

V1 不支持现货、不支持期权、不支持双向持仓模式，也不处理与这些模式相关的特有规则。

#### 2.3.4 多用户与角色权限体系

V1 面向个人使用，不设计多用户账号体系、角色权限划分、用户间数据隔离、代理操作或多级审批流程。

#### 2.3.5 跨频道协同决策

虽然多个频道可对同一标的产生信号，但系统不会在 V1 中建立跨频道优先级、冲突消解或合并分析机制。各频道只在自己的上下文和虚拟子仓位范围内做判断。

### 2.4 已确认业务边界

以下边界视为 V1 固定规则，除非后续评审明确修改，否则按此执行：

- 系统定位为个人自用系统，仅考虑单操作人，不设计多用户、多角色和复杂权限体系。
- 交易所仅为 OKX。
- 交易品类仅为合约 / 永续合约。
- 账户类型为单账户。
- 持仓模式为单向。
- 保证金模式为逐仓。
- 默认杠杆为可配置，初始默认值为 `25x`。
- Telegram 默认接入方式为网页抓取，默认抓取频率为 `30s`。
- 默认订单类型为限价单；频道明确要求市价时，允许市价执行。
- 对 OKX 未上线的标的直接跳过，不进入下单流程。
- 未成交订单在 `5h` 后自动撤销。
- 新开仓低置信度进入人工确认。
- 跟进动作默认不因低置信度而被阻塞，但必须受策略边界约束。
- 新开仓资金投入比例由 AI 在可用资金 `40%-80%` 范围内决策，该能力保留为 V1 可配置策略。
- OpenClaw 属于 V1 交付范围。

### 2.5 范围控制原则

在详细设计和开发阶段，如出现新的需求点，应按以下原则判断是否纳入 V1：

- 如果该需求不影响最小闭环成立，则优先延后。
- 如果该需求显著增加外部依赖、接入复杂度或合规风险，则优先延后。
- 如果该需求会改变单账户、单向持仓、频道隔离这三项核心假设，则默认不纳入 V1。
- 如果该需求引入多用户、多角色、复杂权限或审批体系，则默认不纳入 V1。
- 如果该需求仅用于提升体验，但不影响核心链路打通，可记录到 V1.1 / V1.5 待办。

## 3. 使用者与使用场景

### 3.1 使用者定义

V1 面向单一系统所有者 / 使用者。系统默认只有一个实际操作者，该使用者可能在不同时间承担配置、观察、人工确认和日常运维职责，但这些职责是同一个人的不同操作视角，而不是系统中的多个账号或角色。

该使用者在 V1 中通常需要完成以下事情：

- 配置和维护频道列表
- 配置抓取参数、模型参数和默认交易参数
- 查看消息解析结果、执行结果、持仓、挂单和盈亏
- 对低置信度新开仓动作进行人工确认
- 处理异常日志、切换运行环境和执行必要的低风险控制动作
- 通过 OpenClaw 查询状态并在必要时发起控制操作

为了简化 V1 设计，系统明确不考虑以下能力：

- 多用户账号体系
- 角色权限划分
- 不同用户之间的数据隔离与审批流转
- 代理操作、交接班和多级确认流程

### 3.2 典型使用场景

#### 3.2.1 场景一：新增一个频道并开始监控

使用者在 Web 控制台中新增一个 Telegram 公开频道，配置抓取方式、抓取状态和启停开关。系统保存该频道配置后，应在不重启服务的前提下将该频道加入轮询列表，并开始维护该频道的抓取 cursor、原始消息、标准化消息和版本记录。

#### 3.2.2 场景二：频道发出新的开仓信号

某频道发布一条新的开仓信号。系统识别为新消息后，需读取该频道最近相关消息和当前虚拟子仓位状态，构建上下文并交给 AI 决策。若模型对新开仓动作置信度足够，则直接进入执行链路；若置信度不足，则进入人工确认队列，等待使用者确认后再执行。

#### 3.2.3 场景三：频道对已有仓位发出跟进消息

某频道对已存在的虚拟子仓位发出后续说明，例如上移止损、部分止盈或撤销未成交挂单。系统应将该消息与当前虚拟持仓关联，结合上下文进行 AI 解析，并在策略允许范围内自动执行相关管理动作，不因“低置信度但已持仓”而默认阻塞。

#### 3.2.4 场景四：同一标的被多个频道同时操作

两个不同频道可能同时对同一标的、同一方向给出有效信号。系统应允许这两个频道在内部各自形成独立的虚拟子仓位记录。真实账户层面只存在单一净仓位，但系统必须在内部保留“频道动作 -> 真实执行”的映射关系，以便后续解释盈亏、跟进动作和审计记录。

#### 3.2.5 场景五：使用者通过控制台查看系统状态

使用者需要在 Web 控制台中查看近期日志、当前挂单、实际持仓、频道虚拟子仓位和 PnL。对于低置信度新开仓信号，使用者需要能快速看到消息原文、上下文摘要、AI 决策结果和人工确认入口。

#### 3.2.6 场景六：通过 OpenClaw 查询状态和执行控制

使用者在 Telegram 群组 topic 中通过 OpenClaw 查询系统状态，例如“最近某频道有哪些动作”“当前某标的有哪些虚拟持仓”“系统是否开启自动交易”。对于暂停频道、恢复频道等低风险动作，系统可以直接处理；对于切换实盘、强平、手动开仓等高风险动作，系统必须要求二次确认并保留审计记录。

### 3.3 使用者目标总结

综合以上使用场景，V1 应满足以下目标：

- 使用者可以用尽量少的配置成本启动并管理整个系统。
- 使用者可以在一个控制台里完成观察、确认和日常运维。
- 系统只在真正需要人工判断的新开仓场景打断自动链路。
- 使用者可以通过 Web 和 OpenClaw 两种方式快速查看状态和介入处理。

## 4. 业务规则与核心约束

### 4.1 频道隔离规则

频道隔离是 V1 的核心设计前提之一。系统必须将频道作为上下文边界、判断边界和虚拟子仓位归属边界。

具体要求如下：

- AI 输入上下文只能来自同一个频道，不得混入其他频道的消息。
- 某频道的未完成信号、确认状态、虚拟持仓、挂单状态不得被其他频道直接复用。
- 即使两个频道讨论同一标的，系统也应在内部将其视为两个独立的交易叙事。
- 真实账户的净仓位变化必须可追溯回一个或多个频道的动作映射，但映射关系不能反过来打破频道独立性。

该规则的目的，是保证系统在面对多频道同时发信号时仍然具备解释性和可审计性。

### 4.2 人工确认规则

V1 的人工确认机制只对“新开仓低置信度”生效，这是已经确认的业务边界。设计理由是：

- 新开仓是新增风险暴露的动作，一旦判断错误，直接带来新增敞口。
- 已有仓位的跟进动作通常属于风险管理或仓位调整，若一律阻塞，反而可能因为延误操作造成更大损失。
- 因此 V1 的确认目标不是“控制所有风险”，而是“优先拦截新增风险暴露”。

具体要求如下：

- 当 AI 输出的动作为新开仓，且置信度低于阈值时，系统必须进入人工确认队列。
- 当 AI 输出的动作为补仓、减仓、平仓、止损调整、止盈调整、撤单或改单等跟进动作时，系统默认不因置信度不足而阻塞。
- 确认记录必须保留消息来源、上下文摘要、AI 结果、人工结论、处理时间和操作入口。
- 确认通过后，应继续进入执行链路；确认拒绝后，应记录原因并结束该次执行请求。

### 4.3 交易执行规则

V1 的交易执行规则必须围绕 OKX 合约执行能力来设计，核心原则是“优先映射为 OKX 支持的稳定动作模型”。

具体要求如下：

- 默认订单类型为限价单。
- 当频道文本中明确要求市价时，系统可使用市价单。
- 当频道给出入场区间时，AI 可在策略范围内决定在区间内如何挂单或拆单。
- 对未在 OKX 上线的标的，系统必须直接跳过，不生成执行任务。
- 对未成交挂单，系统必须在 `5h` 后自动撤销，除非此前已被消息显式取消或已完成状态变化。
- 对“止损、止盈、多个止盈点、分批止盈、移动到保本、撤单、改单、reduce-only 平仓”等管理动作，执行层必须负责把意图拆分成 OKX 可支持的订单模型。

### 4.4 风险与资金规则

V1 明确保留较高风险的资金分配策略，但必须把相关状态计算和控制能力做完整。

具体要求如下：

- 系统必须实时或准实时掌握账户可用余额。
- 系统必须跟踪每个频道虚拟子仓位已占用或已预留的保证金。
- 当发生新开仓动作时，AI 可以在当前钱包可用资金的 `40%-80%` 范围内决定本次投入比例。
- 由于该比例本身风险较高，系统必须将其设计为可配置项，而不是写死逻辑。
- 模拟盘与实盘必须隔离，任何与真实资金相关的关键切换动作都必须具备明确确认机制和审计记录。

### 4.5 真实仓位与虚拟子仓位约束

由于 OKX 单账户 + 单向持仓模式下只能看到真实净仓位，系统无法在交易所原生层面区分“这部分仓位属于哪个频道”。因此 V1 必须通过虚拟子仓位账本补足这一层抽象。

具体要求如下：

- 系统内部必须按频道维护虚拟持仓、虚拟均价、挂单、成交和盈亏估算。
- 执行层在提交 OKX 真实订单时，必须保留虚拟动作与真实订单之间的映射关系。
- 当多个频道共同影响同一真实净仓位时，系统仍需能解释每个频道当前理论持仓状态。
- 后续跟进动作判断必须优先依据频道自己的虚拟状态，而不是简单依据真实净仓位总量。

### 4.6 安全与可审计性约束

V1 在具备自动执行能力的同时，必须满足基础安全和审计要求。

具体要求如下：

- 登录密码必须强哈希存储，不得明文保存。
- API Key、Telegram 凭据、模型密钥等敏感配置不得出现在前端展示和普通日志中。
- 关键事件必须带时间戳和关联 ID。
- 人工确认、高风险控制动作、环境切换、手动干预等操作必须具备审计记录。
- 即使系统仅供单人使用，也应能区分系统自动动作与人工动作。
- OpenClaw 和 Web 控制台对高风险动作的处理边界必须保持一致。

## 5. 功能需求

### 5.1 频道管理

频道管理模块负责定义系统的监控对象，是整条业务链路的入口配置层。V1 需要支持在单用户控制台中对频道进行完整的生命周期管理，同时保证历史数据和审计信息不会因为配置变更而丢失。

系统必须支持以下能力：

- 新增频道。至少需要配置频道标识、频道展示名称、频道地址或抓取标识、启停状态和备注信息。
- 编辑频道。支持修改频道名称、抓取参数、启停状态和说明信息，并在保存后运行时生效。
- 停用频道。停用后系统不再继续对该频道发起新的监控与处理，但应保留历史消息、历史决策、历史订单映射和审计记录。
- 删除频道。V1 应采用逻辑删除或等效方式，避免因删除操作造成历史数据不可追溯。
- 频道列表查看。使用者应能看到频道当前状态、最近一次抓取时间、最近一次成功时间、最近一次错误摘要和是否正在参与自动交易。
- 去重校验。同一个频道来源标识不得被重复创建为多个活动中的同类来源配置。
- 热更新。新增、编辑、停用和删除操作应在不重启服务的前提下生效。

系统应同时满足以下约束：

- V1 默认频道来源类型为 Telegram 网页抓取，不要求在界面中提供 Bot API 或 TDLib 配置能力。
- 频道配置变更应写入操作审计，以便后续回溯“谁在什么时候改了什么”。在单人使用场景下，这里的“谁”仍需保留为人工操作标识。
- 频道停用或删除不应自动清空其对应的虚拟子仓位、日志和历史决策数据。

### 5.2 Telegram 消息采集

Telegram 消息采集模块负责从公开频道页面中稳定提取消息变化，是系统的第一道数据入口。V1 仅实现网页抓取模式，因此该模块的目标是尽量稳定、可恢复、可去重地获取可见消息内容。

系统必须支持以下能力：

- 基于 `t.me/s/<channel>` 对公开频道进行轮询抓取。
- 默认轮询周期为 `30s`，并允许后续通过配置进行调整。
- 为每个频道维护独立的抓取 cursor 或等效进度标识，用于识别哪些消息是新发现的、哪些消息是重复抓取的。
- 识别新消息，并将其交给后续标准化与决策链路处理。
- 识别编辑后的消息，并将新旧版本关系交给版本管理模块处理。
- 在可识别范围内记录消息不可用、页面异常或消息缺失等状态变化。
- 保存原始抓取结果中与业务有关的原始内容和基础元信息，便于后续回溯。

采集模块还必须满足以下要求：

- 每个频道的采集状态相互独立，一个频道抓取失败不应阻塞其他频道继续工作。
- 对同一条未发生变化的重复抓取结果，不得重复触发后续决策链路。
- 抓取异常必须记录日志，并保留最近错误信息，便于在控制台中查看。
- 系统重启后，应能恢复各频道的抓取进度，而不是从头重复处理所有历史消息。

### 5.3 消息标准化与版本管理

消息标准化模块负责把采集结果转成系统内部统一可处理的数据结构，同时为消息编辑、重复采集和历史回溯提供基础。V1 的目标不是保留网页展示的全部细节，而是保留足够支撑交易决策和审计的标准化数据。

系统必须支持以下能力：

- 将原始抓取结果标准化为统一消息结构，至少包括频道标识、来源消息标识、抓取时间、正文内容、可见时间信息和版本关系。
- 为同一来源消息维护稳定的内部主键和版本号。
- 区分“首次出现”“内容编辑”“重复无变化”“来源不可用”等处理结果。
- 保存原始消息与标准化消息之间的映射关系。
- 对同一消息的不同版本保留完整历史，便于后续审计和重新决策。
- 在消息被识别为编辑后，触发该消息的新版本进入后续上下文构建与决策流程。

标准化与版本管理还应满足以下约束：

- 标准化结果必须具备幂等性。相同输入不得生成重复的有效标准化记录。
- 同一来源消息的版本号应单调递增，避免出现版本覆盖或顺序混乱。
- 对无法可靠提取的字段，系统应显式标记缺失，而不是静默填入推测值。
- 原始消息记录和版本记录应默认不可变，以便审计时还原真实处理过程。

### 5.4 上下文构建

上下文构建模块负责为 AI 提供足够但不过界的背景信息，是保证决策准确性和频道隔离性的关键模块。V1 中所有上下文都必须严格限定在单频道内。

系统必须支持以下能力：

- 为当前待处理消息收集同频道最近若干条相关消息。
- V1 推荐默认上下文窗口为“同频道最近 `8` 条相关消息”，但该数值应允许使用者调整。
- 注入当前频道尚未完成的信号、未结束的挂单和虚拟子仓位状态。
- 对编辑消息场景，将旧版本信息和新版本信息一起纳入上下文，便于模型判断“是语义修正还是执行意图变更”。
- 对跟进消息场景，将当前频道已有虚拟持仓和最近状态变化一并注入，帮助模型识别其关联对象。
- 为每一次上下文构建生成可持久化的输入快照或等效结构，以便复查“当时模型看到了什么”。

上下文构建必须遵守以下规则：

- 不允许混入其他频道的消息、仓位、挂单、决策或审核信息。
- 当历史上下文不足时，系统仍应生成最小可用上下文，而不是直接跳过处理。
- 若存在多个可能关联的未完成信号，上下文中应完整暴露这些候选项，而不是在 AI 之前进行过度拍板。
- 上下文生成过程应附带关联 ID，便于串联消息、决策、订单和日志。

### 5.5 AI 解析与决策

AI 解析与决策模块负责把自然语言信号转化为结构化动作，是业务链路中的核心判断层。V1 中该模块的重点是结构化、可解释、可审计，而不是完全依赖模型自由发挥。

系统必须支持以下能力：

- 支持在运行时切换模型。
- 支持在运行时切换思考等级。
- 以结构化 JSON 或等效 schema 约束格式返回决策结果。
- 对当前消息和上下文生成动作类型、价格信息、仓位管理意图、置信度和简短理由。
- 在结果中明确标记是否需要人工确认。
- 为每一次 AI 决策保存输入快照、输出结果、模型信息和决策时间。

AI 输出至少应包含以下信息：

- 频道标识和消息标识
- 标的
- 方向
- 动作类型
- 入场方式
- 入场价格或入场区间
- 止损方案或止损调整方案
- 单个或多个止盈目标
- 是否涉及分批止盈、部分减仓、保本移动、撤单或改单
- 置信度
- 简短理由
- 是否需要人工确认

V1 至少应支持以下动作类型：

- `open_position`
- `add_position`
- `reduce_position`
- `close_position`
- `partial_take_profit`
- `set_stop_loss`
- `move_stop_loss`
- `move_stop_to_break_even`
- `set_take_profit`
- `set_multi_take_profit`
- `cancel_orders`
- `replace_orders`
- `force_reduce_only_close`
- `hold`
- `ignore`

AI 决策模块还必须满足以下要求：

- 对新开仓低置信度场景，必须输出“需要人工确认”的结果，而不是直接下单。
- 对已有持仓的跟进消息，即使置信度一般，也应优先给出明确动作或 `hold` 结果，而不是简单回退到人工确认。
- 当缺失关键执行参数且无法从上下文补全时，系统应输出 `hold` 或 `ignore`，不得凭空构造交易参数。
- 对编辑消息和跟进消息，系统应允许重新决策，并保留新旧决策记录。

### 5.6 人工确认

人工确认模块用于承接“新开仓低置信度”场景，是 V1 中唯一正式的人为介入节点。由于系统面向个人使用，该模块的重点不是多级审批，而是单人快速确认和完整审计。

系统必须支持以下能力：

- 为需要确认的新开仓信号生成待确认项。
- 在控制台中展示待确认项列表和详情。
- 对每个待确认项展示消息原文、相关上下文摘要、AI 解析结果、置信度、建议动作和关键价格信息。
- 支持至少两种处理结果：确认执行、拒绝执行。
- 确认执行后，将该决策送入执行规划与 OKX 执行链路。
- 拒绝执行后，记录拒绝结果并终止该次动作，不再自动重试。

人工确认模块还必须满足以下要求：

- 同一待确认项只能被成功执行一次，必须具备幂等保护。
- 若待确认期间原始消息发生编辑、取消或被后续消息显式推翻，系统应能将旧确认项标记为失效或至少在界面中明确提示风险。
- 人工确认记录必须保留确认结果、处理时间、触发来源和后续执行关联关系。
- V1 不设计多人分配、转派、会签或多级审批能力。

### 5.7 OKX 执行

OKX 执行模块负责把结构化动作转为交易所可执行计划，是系统从“理解信号”走向“真实动作”的关键层。V1 只支持 OKX 合约 / 永续合约场景。

系统必须支持以下能力：

- 查询交易标的是否存在以及是否可交易。
- 查询账户可用余额、当前持仓和当前挂单。
- 设置杠杆并按逐仓、单向持仓模式发起执行。
- 提交限价单。
- 在频道明确信号要求市价时提交市价单。
- 撤销挂单。
- 改单或撤单重挂。
- 设置或更新止损。
- 设置或更新止盈。
- 设置多个止盈点。
- 执行部分止盈或分批减仓。
- 执行移动止损和移动到保本。
- 在需要时拆分为条件单、算法单或附带止盈止损单等 OKX 可支持模型。

执行模块必须遵守以下规则：

- 默认执行环境为系统当前选定的模拟盘或实盘环境，二者不得混用凭据和状态。
- 默认订单类型为限价单，除非信号明确要求市价。
- 对 OKX 未上线标的，系统必须直接跳过并记录原因。
- 对一个 AI 动作若需要拆分成多笔真实订单，系统必须记录拆分计划和订单映射关系。
- 每个真实下单请求都必须带有客户端关联标识，用于幂等和审计。
- 未成交订单在 `5h` 后必须自动撤销，除非在此之前已被显式取消、完全成交或被替换。

### 5.8 真实状态同步

真实状态同步模块负责从交易所获取订单、成交、持仓和账户变化，并把这些变化反馈到控制台、日志和虚拟子仓位账本中。该模块的核心目标是保证系统知道“交易所真实发生了什么”。

系统必须支持以下能力：

- 同步真实订单状态变化。
- 同步真实成交结果。
- 同步真实持仓状态变化。
- 同步账户余额和可用资金变化。
- 在订单成交、撤销、部分成交、失败或失效时更新内部状态。
- 将真实状态变化传播到日志、控制台和虚拟子仓位账本。

状态同步模块还必须满足以下要求：

- 系统重启后，应能重新同步当前未完成订单、真实持仓和账户状态。
- 若系统内部预期状态与交易所真实状态不一致，应记录异常并触发补偿同步或提示。
- 同步链路必须能够区分“订单已提交但未成交”“部分成交”“已撤单”“已完全成交”等关键状态。
- 所有状态同步事件应保留时间戳和关联 ID，便于审计和问题排查。

### 5.9 虚拟子仓位账本

虚拟子仓位账本是 V1 的核心差异化能力，用于在 OKX 单账户单向持仓模式下，继续按频道解释和管理交易生命周期。没有这一层，系统将无法正确理解多个频道共用一个真实净仓位时的逻辑归属。

系统必须支持以下能力：

- 为每个频道维护独立的虚拟持仓生命周期。
- 记录每个频道当前虚拟方向、虚拟数量、虚拟均价和当前状态。
- 记录每个频道的挂单映射、成交映射和仓位事件。
- 估算每个频道的已实现盈亏和未实现盈亏。
- 跟踪每个频道已占用或预留的保证金。
- 将 AI 决策、执行计划、真实订单、真实成交与虚拟持仓事件关联起来。

账本模块必须满足以下规则：

- 同一真实净仓位可以映射到多个频道虚拟子仓位，但每个频道的账本状态必须独立计算。
- 某一频道执行减仓或平仓，不应直接覆盖其他频道的虚拟状态。
- 当真实成交结果与预期拆分计划存在差异时，账本应以真实成交为准更新映射，并保留偏差说明。
- 账本数据应具备可重建性，即原则上可以根据历史事件流回放出当前状态。

### 5.10 风控与策略控制

风控与策略控制模块负责集中承载系统的运行边界和执行开关。由于 V1 是个人使用系统，该模块的重点不是复杂权限，而是把关键开关、关键阈值和高风险动作控制集中起来。

系统必须支持以下能力：

- 全局交易开关。
- 频道级启停开关。
- 模拟盘 / 实盘切换。
- 模型切换和思考等级切换。
- 默认杠杆配置。
- 新开仓人工确认阈值配置。
- 允许动作范围或动作白名单配置。
- 新开仓资金投入比例范围配置。

风控与策略控制模块必须满足以下要求：

- 当全局交易开关关闭时，系统仍可继续采集消息、构建上下文、生成决策和记录日志，但不得继续执行真实下单动作。
- 频道级停用至少应阻止该频道继续进入自动执行链路。
- 模拟盘切换到实盘应被视为高风险动作，必须进行明确确认并记录审计日志。
- 关键配置变更应立即生效或在界面中明确提示生效方式，避免使用者误判当前运行状态。
- V1 不需要设计角色权限审批，但必须区分“系统自动行为”和“人工触发行为”。

### 5.11 Web 控制台

Web 控制台是 V1 的主要操作入口，承担配置、观察、确认和个人运维职责。由于系统供单人使用，控制台的重点是信息集中、实时反馈和高风险操作可控，而不是多用户协作。

系统必须支持以下能力：

- 单用户登录。
- `6` 位密码认证。
- 会话管理、登录限流和多次失败临时锁定。
- 频道列表查看和频道配置维护。
- 模型、思考等级、默认杠杆和运行参数配置。
- 模拟盘 / 实盘切换。
- 全局交易开关控制。
- 实时日志查看。
- 挂单、真实持仓、虚拟子仓位和 PnL 查看。
- 人工确认队列查看和处理。
- 系统健康状态查看。

控制台还必须满足以下要求：

- 日志和关键状态应尽量实时刷新。
- 高风险动作应在界面中进行二次确认。
- 敏感配置不得明文展示。
- 控制台中不需要实现用户管理、角色管理或权限配置页面。
- 使用者应能清楚区分真实交易所状态与频道虚拟状态，避免混淆。

### 5.12 OpenClaw 集成

OpenClaw 集成模块用于在 Telegram 协作环境中补充状态查看和有限控制能力。虽然系统供单人使用，但在移动场景或远程场景下，OpenClaw 可以成为轻量级入口。

系统必须支持以下能力：

- 在 Telegram 群组 topic 中输出实时日志摘要。
- 支持自然语言查询系统状态。
- 提供项目专用 skill。
- 支持查询最近日志、当前持仓、当前挂单、PnL 和最近消息解释。
- 支持暂停频道、恢复频道等低风险控制动作。
- 对切换实盘、强平、手动开仓、修改核心风控等高风险动作要求二次确认。

OpenClaw 集成还必须满足以下约束：

- OpenClaw 的高风险动作边界必须与 Web 控制台保持一致。
- OpenClaw 返回内容不得泄露敏感密钥和隐私配置。
- 对于不允许直接执行的动作，系统应明确返回“需要确认”或“当前不允许”的结果，而不是静默失败。
- 所有通过 OpenClaw 发起的查询和控制动作都应写入审计记录。

## 6. 核心流程

本章描述系统在 V1 中必须跑通的关键业务流程。流程设计的目标不是只说明“模块按什么顺序调用”，而是明确每一步的输入、关键判断、分支处理和落库要求，确保后续技术实现、测试验证和问题排查都能对齐同一套流程语义。

### 6.1 新消息处理流程

新消息处理流程适用于某频道出现一条此前尚未被系统处理过的消息。该流程是最基础、也是最频繁触发的主链路。

流程步骤如下：

1. 采集层发现某频道出现新消息，并生成原始抓取记录。
2. 标准化层将该消息转换为统一内部结构，生成标准化消息记录，并判断其为“首次出现”的有效消息。
3. 系统为该消息分配关联 ID，并将频道标识、消息标识、抓取时间等信息写入日志上下文。
4. 上下文构建模块读取同频道最近相关消息、当前未完成信号、当前虚拟子仓位和未结束挂单状态，生成本次 AI 输入快照。
5. AI 决策模块对当前消息和上下文进行结构化解析，输出动作类型、执行参数、置信度、理由和是否需要人工确认。
6. 策略控制模块校验当前环境是否允许执行，包括：
   - 全局交易开关是否开启
   - 当前频道是否启用
   - 标的是否存在于 OKX 可交易范围
   - 是否命中新开仓人工确认阈值
   - 当前环境是模拟盘还是实盘
7. 若 AI 输出结果为 `ignore` 或 `hold`，则流程不进入交易执行，但必须保存决策记录和日志。
8. 若 AI 输出的是新开仓动作且置信度不足，则生成人工确认项，并将该流程转入人工确认队列。
9. 若 AI 输出的是可自动执行动作，则执行规划模块生成 OKX 执行计划，必要时将一个动作拆分为多笔真实订单或附带止盈止损计划。
10. OKX 执行模块提交真实请求，并记录客户端关联标识、执行计划与虚拟动作之间的映射关系。
11. 状态同步模块等待并处理交易所返回的订单、成交和持仓变化，更新真实状态和虚拟子仓位账本。
12. Web 控制台与 OpenClaw 日志输出模块展示本次处理结果，包括成功、失败、跳过、待确认等状态。

该流程必须满足以下要求：

- 新消息从发现到生成决策，必须具备幂等性，避免重复抓取导致重复决策。
- 即使最终未执行交易，也必须保留完整决策记录和处理日志。
- 所有关键节点都应关联同一条主链路 ID，便于事后追踪。

### 6.2 编辑消息处理流程

编辑消息处理流程适用于某条已存在消息在 Telegram 页面中发生了内容修改。由于很多频道会通过编辑消息补充价格、调整止损或直接改写执行意图，因此 V1 必须将编辑事件视为正式业务事件，而不是简单覆盖旧文本。

流程步骤如下：

1. 采集层发现某条已知消息内容发生变化，并生成新的原始抓取记录。
2. 标准化层识别该消息为已有消息的新版本，而非一条全新的消息。
3. 系统保存旧版本与新版本内容，并为该消息版本号递增。
4. 系统记录本次事件为“编辑消息处理”，并将旧版本摘要、新版本摘要和关联 ID 放入日志上下文。
5. 上下文构建模块读取：
   - 当前新版本消息
   - 该消息的旧版本内容
   - 同频道最近相关消息
   - 当前虚拟子仓位和未完成挂单状态
6. AI 决策模块重新判断该编辑行为的真实意图，典型结果包括：
   - 仅文本修正，不改变交易意图
   - 调整开仓价格或开仓区间
   - 撤销原有挂单
   - 调整止损或止盈
   - 修改已有执行计划
7. 若 AI 判断该编辑仅为文字润色或无实质含义变化，则仅记录新版本和决策结果，不触发交易动作。
8. 若 AI 判断该编辑改变了已有执行意图，则执行规划模块根据“旧状态与新意图差异”生成补救动作，例如撤单、改单、更新止损或更新止盈。
9. 若该编辑涉及尚未执行的新开仓且新版本仍为低置信度，则应保持或更新待确认项，而不是直接执行。
10. 若该编辑推翻了一个尚在待确认中的旧信号，系统应将旧确认项标记为失效、已过期或需要重新确认。
11. 若该编辑影响了已在交易所挂出的订单，系统必须记录“哪一个编辑消息导致了哪一次执行变更”。

该流程必须满足以下要求：

- 旧版本内容不得被覆盖丢失。
- 编辑处理的核心是“按差异做动作”，而不是简单按新文本重新从零执行。
- 若系统无法安全判断编辑含义，应倾向于记录并提示，而不是冒进执行高风险变更。

### 6.3 跟进消息处理流程

跟进消息处理流程适用于某频道在已有交易叙事基础上继续发出新的说明或指令。该流程与新消息流程的主要区别在于：它通常面向一个已经存在的频道虚拟子仓位或未完成信号，因此判断重点是“这条消息在管理什么”。

流程步骤如下：

1. 采集层识别到一条新的频道消息，并完成标准化。
2. 上下文构建模块优先读取该频道当前活跃虚拟子仓位、最近挂单、最近成交和近期消息链路。
3. 系统尝试识别这条消息可能关联的对象，包括某个已有虚拟持仓、某组挂单或某次未完成交易计划。
4. AI 决策模块根据上下文判断这条消息属于以下哪类：
   - 继续持有
   - 补仓
   - 部分止盈
   - 移动止损
   - 移动到保本
   - 平仓
   - 撤销挂单
   - 修改挂单
   - 无效说明或纯评论
5. 若 AI 输出结果为 `hold` 或 `ignore`，系统保存决策结果即可，不再进入执行。
6. 若 AI 输出结果为有效跟进动作，则流程默认不进入人工确认，而是直接进入执行规划。
7. 执行规划模块根据当前虚拟子仓位状态和真实订单状态，生成需要提交给 OKX 的动作。
8. OKX 执行模块提交实际操作，并将本次跟进消息与真实订单、真实成交和虚拟仓位事件建立映射关系。
9. 状态同步模块在交易所回报到达后，更新真实状态和频道级虚拟子仓位账本。
10. 审计日志必须明确记录“哪条跟进消息触发了哪次状态变化”。

该流程必须满足以下要求：

- 跟进消息不得被误判为全新开仓而导致重复新增风险暴露。
- 即使同一标的存在多个频道持仓，当前跟进消息也只能关联本频道上下文。
- 若当前频道不存在可合理关联的活跃对象，系统应输出 `hold`、`ignore` 或异常提示，而不是强行套用到其他频道或其他历史动作上。

### 6.4 人工确认流程

人工确认流程只用于承接“新开仓低置信度”场景，是自动链路中的单一人工中断点。由于系统是个人使用场景，该流程的目标是快速确认、避免漏判，并确保结果具备完整记录。

流程步骤如下：

1. 某条新开仓决策因置信度不足或命中确认阈值，被系统标记为“需要人工确认”。
2. 系统生成待确认项，并写入确认队列。
3. 待确认项至少应包含以下信息：
   - 频道和消息标识
   - 消息原文与上下文摘要
   - AI 决策结果与关键价格参数
   - 置信度与简短理由
   - 当前环境信息，例如模拟盘或实盘
4. 使用者在 Web 控制台查看待确认项详情。
5. 使用者可选择“确认执行”或“拒绝执行”。
6. 若选择确认执行：
   - 系统先检查该确认项是否仍然有效
   - 若消息未被后续编辑推翻、未被显式取消、未重复执行，则进入执行规划和 OKX 执行流程
   - 执行结果回写到确认记录中
7. 若选择拒绝执行：
   - 系统记录拒绝结果和处理时间
   - 该确认项状态终止，不再继续自动执行
8. 若在待确认期间消息发生编辑、频道被停用、全局交易被关闭或环境发生关键变化，系统应在确认前再次校验是否允许执行。
9. 所有确认动作都必须写入审计日志，并与后续真实执行结果建立关联。

该流程必须满足以下要求：

- 同一确认项只能成功执行一次。
- 失效确认项不得在旧页面状态下被误点后继续下单。
- 确认流程不应被扩展为多级审批或多用户协作流程。

### 6.5 环境切换与高风险确认流程

环境切换与高风险确认流程用于处理那些不会频繁发生、但一旦执行会显著改变系统风险边界的操作。V1 中最核心的高风险动作是从模拟盘切换到实盘，此外还包括强平、手动开仓和核心风控参数变更等控制动作。

流程步骤如下：

1. 使用者在 Web 控制台或 OpenClaw 中发起某项高风险操作，例如切换到实盘。
2. 系统识别该操作属于高风险动作，不允许直接静默生效。
3. 系统向使用者展示或返回确认信息，至少应说明：
   - 当前环境状态
   - 目标状态
   - 该操作可能带来的直接后果
   - 是否存在未完成订单、未处理确认项或当前持仓
4. 使用者再次确认后，系统才允许继续执行该操作。
5. 系统在真正切换前执行前置校验，例如：
   - 配置是否完整
   - 所需凭据是否可用
   - 当前是否存在不适合切换的未完成状态
6. 校验通过后，系统更新当前环境状态，并将结果同步到控制台与日志系统。
7. 若校验失败或使用者取消，系统不得发生实际切换，但必须保留日志记录。
8. 对于强平、手动开仓、修改核心风控等其他高风险动作，也应遵循同样的“识别高风险 -> 展示后果 -> 二次确认 -> 校验执行 -> 写入审计”的模式。

该流程必须满足以下要求：

- 高风险动作的确认边界在 Web 控制台和 OpenClaw 中必须一致。
- 环境切换不得因为界面刷新延迟或重复点击而发生多次实际变更。
- 高风险动作的确认和执行结果都必须具备完整审计记录。

## 7. 数据对象与状态设计

本章定义 V1 的核心数据对象、关键标识和状态流转要求。数据设计的目标不是为了把所有字段一次性定死，而是先明确哪些对象必须存在、各对象分别承担什么职责、对象之间如何关联，以及哪些状态变化必须能被完整追溯。

V1 的数据设计必须遵守以下原则：

- 以“可追溯”为优先。关键对象原则上都应保留创建时间、更新时间、关联 ID、环境标识和必要的来源信息。
- 以“事件可回放”为优先。尤其是消息版本、订单变化、成交变化、虚拟子仓位事件和人工确认记录，不应只保留最终结果。
- 以“真实状态与虚拟状态分层”为优先。交易所真实状态和频道虚拟状态必须分别建模，不能混为一层。
- 以“单用户自用”为前提。V1 不需要 `users`、`roles`、`permissions` 等多用户权限表。

### 7.1 核心数据对象

V1 的核心数据对象可分为六组：配置与来源对象、消息对象、决策与人工确认对象、交易所对象、虚拟账本对象，以及日志与设置对象。

#### 7.1.1 配置与来源对象

##### `channels`

`channels` 用于保存频道主记录，是频道级隔离的根对象。每条频道记录至少应表达以下含义：

- 频道内部唯一标识
- 频道展示名称
- 频道来源标识，例如公开频道路径或名称
- 当前启停状态
- 备注、标签或说明信息
- 创建时间、更新时间、逻辑删除状态

该对象必须满足以下要求：

- 一个频道只代表一个业务隔离单元。
- 频道停用或删除不得影响其历史消息、历史决策和历史账本记录的可追溯性。
- 频道状态变更必须可审计。

##### `channel_sources`

`channel_sources` 用于保存频道接入来源配置。虽然 V1 只实现网页抓取，但该对象仍应保留来源层抽象，以便后续扩展。

该对象至少应描述：

- 所属 `channel_id`
- 来源类型，例如 `telegram_web`
- 来源地址、抓取参数或附加配置
- 当前来源状态
- 最近一次成功时间和最近错误摘要

该对象必须满足以下要求：

- 一个频道在 V1 中通常只会有一个有效网页来源，但结构上允许未来扩展。
- 来源对象与频道主对象应分离，避免将抓取实现细节写死在频道主表中。

##### `source_cursors`

`source_cursors` 用于保存每个来源当前抓取进度，是重启恢复和去重的重要基础。

该对象至少应描述：

- 所属来源标识
- 最近一次已处理的消息进度
- 最近抓取时间、最近成功时间
- 最近错误时间和错误摘要

该对象必须满足以下要求：

- 每个来源的 cursor 独立维护。
- cursor 更新必须是幂等且可恢复的。
- 抓取失败不能把进度错误推进到不可回滚状态。

#### 7.1.2 消息对象

##### `raw_messages`

`raw_messages` 用于保存采集层得到的原始抓取结果，是最接近来源页面的原始证据。

该对象至少应描述：

- 所属 `channel_id`
- 来源消息标识
- 原始正文或原始片段
- 抓取时间
- 本次抓取对应的来源快照信息
- 该次抓取是否识别到变化

该对象必须满足以下要求：

- 原始记录默认不做覆盖更新，避免丢失抓取现场。
- 即使标准化失败，原始记录仍应保留，便于后续排查。

##### `normalized_messages`

`normalized_messages` 用于保存系统内部统一格式的消息对象，是后续上下文构建和 AI 决策的基础输入。

该对象至少应描述：

- 所属 `channel_id`
- 对应来源消息标识
- 当前标准化正文
- 消息可见时间或来源时间
- 当前版本号
- 当前处理状态，例如首次出现、编辑、重复无变化、不可用
- 是否已经进入后续决策流程

该对象必须满足以下要求：

- 同一来源消息应有稳定的内部归属关系。
- 标准化结果应能定位回原始抓取记录。
- 当前版本信息和历史版本信息必须可区分。

##### `message_versions`

`message_versions` 用于保存同一消息的版本历史，是处理编辑消息的关键对象。

该对象至少应描述：

- 所属标准化消息
- 来源版本号或内部版本号
- 该版本正文
- 版本创建时间
- 该版本与上一版本的差异摘要或可推导差异关系

该对象必须满足以下要求：

- 版本记录应按时间或版本号严格递增。
- 任意时点都应能还原某条消息的历史文本变化。
- 编辑消息触发的后续决策应能明确引用“基于哪个版本”。

#### 7.1.3 决策与人工确认对象

##### `ai_decisions`

`ai_decisions` 用于保存模型输出的结构化决策结果，是连接消息、执行和账本的核心桥梁对象。

该对象至少应描述：

- 所属 `channel_id`
- 所属消息和消息版本
- AI 输入快照或其引用
- 模型名称和思考等级
- 动作类型
- 关键执行参数
- 置信度和简短理由
- 是否需要人工确认
- 决策状态
- 决策时间

该对象必须满足以下要求：

- 一次有效 AI 调用至少对应一条决策记录。
- 同一消息在不同版本、不同时间、不同上下文下可产生多条决策记录。
- `ai_decisions` 必须能够回溯到它所基于的消息版本和上下文快照。
- 若实现上不单独建 `context_snapshots` 表，则至少应在该对象中持久化输入快照。

##### `manual_reviews`

为与前期说明文档保持兼容，V1 可继续沿用 `manual_reviews` 这一对象名，但其业务语义在本项目中等同于“人工确认记录”。

该对象至少应描述：

- 所属 `decision_id`
- 确认状态，例如待确认、已确认、已拒绝、已失效
- 确认时间或失效时间
- 确认时展示给使用者的关键摘要
- 后续是否已触发执行

该对象必须满足以下要求：

- 一个需要人工确认的决策，原则上最多对应一个当前有效确认记录。
- 确认结果必须与后续执行结果建立关联。
- 被编辑消息推翻或被后续动作覆盖的旧确认项必须能够标记失效。

#### 7.1.4 交易所对象

##### `exchange_orders`

`exchange_orders` 用于保存系统向 OKX 提交的真实订单和订单计划映射结果。

该对象至少应描述：

- 所属 `decision_id`
- 所属 `virtual_position_id` 或对应虚拟事件引用
- 本地客户端订单标识 `cl_ord_id`
- 交易所订单标识 `okx_ord_id`
- 交易标的、方向、价格、数量、订单类型
- 当前订单状态
- 提交时间、更新时间、取消时间或完成时间
- 所属环境，例如模拟盘或实盘

该对象必须满足以下要求：

- 一个 AI 决策可以拆分成多笔真实订单，因此 `ai_decisions` 到 `exchange_orders` 应允许一对多关系。
- 订单状态必须以交易所真实回报为准持续更新。
- 对撤单、改单、替换等动作，应保留新旧订单之间的关系。

##### `exchange_fills`

`exchange_fills` 用于保存真实成交记录，是驱动账本变化的最小可信执行单元。

该对象至少应描述：

- 所属真实订单
- 成交数量、成交价格、成交时间
- 手续费或相关成交成本信息
- 成交方向和成交类型

该对象必须满足以下要求：

- 每笔真实成交都应独立保存，不得只累计到订单表中。
- 账本更新必须能够追溯到成交级别，而不是只追溯到订单级别。

##### `exchange_position_snapshots`

`exchange_position_snapshots` 用于保存某一时点交易所真实持仓快照。

该对象至少应描述：

- 快照时间
- 标的
- 方向
- 当前数量
- 开仓均价或持仓均价
- 未实现盈亏等关键状态
- 所属环境

该对象必须满足以下要求：

- 快照对象用于恢复和核对真实状态，不替代订单与成交事件。
- 当系统重启或发现状态不一致时，应能依赖快照做恢复或校验。

##### `account_snapshots`

`account_snapshots` 用于保存账户余额和可用资金快照。

该对象至少应描述：

- 快照时间
- 总权益
- 可用余额
- 已占用保证金
- 所属环境

该对象必须满足以下要求：

- 新开仓资金比例计算必须能够引用最近有效账户快照。
- 模拟盘与实盘账户快照必须严格隔离。

#### 7.1.5 虚拟账本对象

##### `virtual_positions`

`virtual_positions` 用于保存频道级虚拟子仓位主记录，是频道交易生命周期的承载对象。

该对象至少应描述：

- 所属 `channel_id`
- 标的和方向
- 当前虚拟数量
- 当前虚拟均价
- 当前生命周期状态
- 起始时间、结束时间
- 当前已实现盈亏、未实现盈亏估算
- 当前已占用或预留保证金

该对象必须满足以下要求：

- 一个频道在同一交易叙事中应有稳定的 `virtual_position_id`。
- 虚拟子仓位状态必须与真实订单和真实成交保持可映射关系，但不能直接被真实净仓位覆盖。
- 当虚拟数量回到零且生命周期结束后，该对象应进入终态，而不是被硬删除。

##### `virtual_position_events`

`virtual_position_events` 用于保存虚拟子仓位的事件流，是账本可回放的核心。

该对象至少应描述：

- 所属 `virtual_position_id`
- 事件类型，例如开仓计划、首次成交、补仓、减仓、止盈、止损、撤单、关闭
- 事件时间
- 事件对应的决策、订单或成交引用
- 事件前后关键数量或均价变化

该对象必须满足以下要求：

- 虚拟持仓当前状态原则上应可通过事件回放重建。
- 事件对象必须能够表达“计划动作”和“实际成交结果”的区别。
- 对同一频道的每次仓位变化，都应有明确事件记录。

##### `pnl_snapshots`

`pnl_snapshots` 用于保存盈亏快照，既可按频道维度，也可按账户维度聚合。

该对象至少应描述：

- 快照时间
- 所属频道或所属虚拟持仓
- 已实现盈亏
- 未实现盈亏
- 统计口径说明

该对象必须满足以下要求：

- 盈亏快照是展示和统计对象，不替代事件账本。
- 盈亏口径必须稳定，避免前后展示数字无法解释。

#### 7.1.6 日志、操作与设置对象

##### `system_logs`

`system_logs` 用于保存系统运行日志和关键业务日志。

该对象至少应描述：

- 日志时间
- 日志级别
- 模块来源
- 关联 ID
- 事件摘要
- 脱敏后的上下文信息

该对象必须满足以下要求：

- 日志必须支持按关联 ID 追踪整条链路。
- 敏感配置不得以明文形式写入日志。
- 日志既要支持问题排查，也要支持控制台实时展示。

##### `operator_actions`

`operator_actions` 用于保存人工触发的操作记录。虽然系统是单用户场景，但仍然需要显式区分“系统自动动作”和“人工动作”。

该对象至少应描述：

- 操作时间
- 操作类型
- 触发入口，例如 Web 或 OpenClaw
- 目标对象，例如频道、配置项、确认项、订单或环境开关
- 操作结果
- 关联 ID

该对象必须满足以下要求：

- 高风险动作必须落库到该对象或等效审计对象。
- 人工动作必须能追溯到其后续影响的业务对象。

##### `settings`

`settings` 用于保存运行时全局配置和默认参数。

该对象至少应描述：

- 当前环境标识
- 全局交易开关状态
- 默认杠杆
- 模型选择
- 思考等级
- 新开仓人工确认阈值
- 资金投入比例范围
- 其他基础策略配置

该对象必须满足以下要求：

- 关键设置变更必须可审计。
- 模拟盘与实盘相关配置不得混存为一组不可区分数据。
- 配置值应支持热更新和运行时读取。

#### 7.1.7 对象关系要求

除单个对象职责外，V1 还必须满足以下对象关系：

- 一个 `channel` 对应多个消息、多个决策、多个虚拟子仓位事件和多条日志。
- 一条 `normalized_message` 可对应多个 `message_versions`，也可对应多条 `ai_decisions`。
- 一条 `ai_decision` 可对应零或一条当前有效的 `manual_reviews`，以及零到多条 `exchange_orders`。
- 一条 `exchange_order` 可对应零到多条 `exchange_fills`。
- 一个 `virtual_position` 可对应多条 `virtual_position_events`、多条 `exchange_orders` 和多条 `pnl_snapshots`。
- `system_logs` 与 `operator_actions` 应可通过 `correlation_id` 或等效关联键串联到业务主链路。

### 7.2 关键标识

V1 的关键标识设计必须满足“内部稳定、外部可映射、链路可串联、环境可区分”四项要求。

#### 7.2.1 `channel_id`

`channel_id` 是系统内部的频道主标识。

要求如下：

- 在系统内部全局唯一且稳定。
- 不因频道名称修改、启停或抓取参数调整而变化。
- 所有与频道强关联的业务对象都应能回溯到该标识。

#### 7.2.2 `source_message_id`

`source_message_id` 是来源侧消息标识，用于关联同一条 Telegram 消息。

要求如下：

- 在同一频道来源范围内应保持稳定。
- 必须和 `channel_id` 组合使用，避免不同频道间碰撞。
- 不能替代系统内部主键，但必须原样保留以支持回溯。

#### 7.2.3 `source_edit_version`

`source_edit_version` 或等效内部版本号用于描述同一来源消息的版本递增关系。

要求如下：

- 在同一条来源消息内部单调递增。
- 必须能区分初始版本与后续编辑版本。
- 应与 `channel_id + source_message_id` 共同构成消息版本唯一性约束的基础。

#### 7.2.4 `decision_id`

`decision_id` 是 AI 决策记录的内部唯一标识。

要求如下：

- 每次有效 AI 输出都应生成独立 `decision_id`。
- `decision_id` 必须能关联到消息版本、人工确认记录、执行计划和订单映射。
- 同一消息多次决策不得复用同一个 `decision_id`。

#### 7.2.5 `virtual_position_id`

`virtual_position_id` 是频道级虚拟子仓位的主标识。

要求如下：

- 一个虚拟持仓生命周期内保持稳定。
- 不因期间出现补仓、减仓、移动止损或分批止盈而变化。
- 所有虚拟仓位事件、订单映射和盈亏快照都应能追溯到该标识。

#### 7.2.6 `cl_ord_id`

`cl_ord_id` 是系统生成的客户端订单标识，用于提交给 OKX 并实现幂等控制。

要求如下：

- 每个真实下单请求都必须带有唯一 `cl_ord_id`。
- 一旦请求提交，不应在不同订单上复用。
- 应能反向映射回 `decision_id` 和 `virtual_position_id`。

#### 7.2.7 `okx_ord_id`

`okx_ord_id` 是交易所返回的真实订单标识。

要求如下：

- 必须原样保存。
- 必须与本地 `cl_ord_id` 建立稳定映射。
- 所有订单状态同步、撤单、改单和成交处理都应优先依赖该标识完成对账。

#### 7.2.8 `correlation_id`

`correlation_id` 是贯穿整条业务链路的关联标识，用于把采集、决策、执行、日志和审计串到一起。

要求如下：

- 每条主要处理链路都应生成主 `correlation_id`。
- 同一条链路中的原始消息、决策、确认、订单、日志和人工操作应共享或派生自同一主关联标识。
- 在问题排查和控制台日志检索时，应支持按 `correlation_id` 聚合查看。

#### 7.2.9 其他通用标识要求

除以上显式标识外，V1 还应满足以下通用要求：

- 所有核心对象都应具备内部主键。
- 所有时间字段应统一时区策略和存储口径。
- 涉及模拟盘 / 实盘的数据对象都应具备环境区分字段，避免数据串用。
- 对需要幂等的对象，应设计唯一键或等效幂等约束。

### 7.3 关键状态流转

状态流转设计的目标，是让系统能回答“当前走到哪一步了”“为什么停在这里”“下一步会发生什么”。V1 至少需要明确五类关键状态流转。

#### 7.3.1 消息状态流转

消息对象至少应覆盖以下流转语义：

1. 原始消息被抓取并保存为 `captured`。
2. 标准化成功后进入 `normalized`。
3. 然后根据识别结果分流为：
   - `new_message`
   - `edited_message`
   - `duplicate_unchanged`
   - `source_unavailable`
4. 对 `new_message` 和 `edited_message`，进入上下文构建与决策链路。
5. 决策完成后，消息处理结果可继续落为：
   - `decision_generated`
   - `decision_skipped`
   - `needs_confirmation`
   - `execution_triggered`
   - `completed`

要求如下：

- `duplicate_unchanged` 不应重复触发决策。
- `edited_message` 必须能定位到前一版本。
- 消息状态变化必须可追踪到具体处理时间和关联 ID。

#### 7.3.2 决策状态流转

`ai_decisions` 至少应覆盖以下流转语义：

1. 决策创建时进入 `generated` 或等效初始状态。
2. 然后根据结果进入以下分支之一：
   - `hold`
   - `ignore`
   - `needs_confirmation`
   - `ready_for_execution`
3. 若进入 `needs_confirmation`，后续根据人工结果流转为：
   - `confirmed`
   - `rejected`
   - `invalidated`
4. 若进入 `ready_for_execution` 或 `confirmed`，则继续流转为：
   - `execution_planned`
   - `execution_submitted`
   - `execution_synced`
   - `execution_failed`

要求如下：

- 决策状态应能区分“模型已生成结论”与“交易已真正提交”。
- 同一决策进入终态后，不应被旧流程重复推进。
- 决策状态与订单状态、确认状态要保持可对账关系。

#### 7.3.3 人工确认状态流转

人工确认对象至少应覆盖以下流转语义：

1. 创建后进入 `pending`。
2. 使用者处理后进入以下终态之一：
   - `approved`
   - `rejected`
   - `expired`
   - `invalidated`
3. `approved` 后可继续带出一个“已触发执行”的衍生状态或执行引用。

要求如下：

- `pending` 项必须可识别是否仍然有效。
- 被消息编辑推翻或被后续动作覆盖的确认项应进入 `invalidated` 或等效状态。
- 一个确认项只能产生一次成功执行结果。

#### 7.3.4 订单状态流转

真实订单对象至少应覆盖以下流转语义：

1. 执行规划生成后进入 `planned`。
2. 成功向 OKX 提交后进入 `submitted`。
3. 随着交易所回报继续流转为：
   - `live`
   - `partially_filled`
   - `filled`
   - `canceled`
   - `replaced`
   - `rejected`
   - `expired`

要求如下：

- 本地状态必须能够映射到交易所真实状态。
- `partially_filled` 必须能触发增量账本更新，而不是只等到完全成交。
- 撤单、改单和替换关系必须在状态与对象关系上都能体现。

#### 7.3.5 虚拟子仓位生命周期流转

虚拟子仓位对象至少应覆盖以下生命周期语义：

1. 新开仓决策成立后，可创建一个 `initiated` 或等效的虚拟持仓起点。
2. 首笔开仓订单提交但尚未形成完整仓位时，进入 `opening`。
3. 一旦产生有效持仓，进入 `open`。
4. 在后续补仓、减仓、止盈、止损或移动保本过程中，可处于 `adjusting` 或等效中间状态。
5. 当虚拟数量回到零并结束交易叙事后，进入 `closed`。
6. 若开仓计划在未形成实际持仓前被取消，也可进入 `canceled` 或等效终态。

要求如下：

- 生命周期状态应反映频道级持仓叙事，而不是简单镜像真实净仓位。
- 虚拟子仓位关闭后，其历史事件和盈亏记录必须保留。
- 对同一频道的新一轮独立交易叙事，应新建新的 `virtual_position_id`，而不是复用旧生命周期。

## 8. 非功能需求

本章定义 V1 在功能之外必须满足的系统性要求。对于本项目来说，非功能需求并不是“锦上添花”，而是决定系统是否能够长期稳定运行、是否可放心切换到实盘、以及出现问题时是否还能快速定位和恢复的基础。

V1 的非功能设计应遵循以下总体原则：

- 优先保证资金安全和操作边界清晰。
- 优先保证消息处理、决策处理和执行处理具备幂等性。
- 优先保证系统出了问题之后可以定位、可以恢复、可以解释。
- 性能目标以“满足个人日常使用和频道跟单场景”为主，不追求过度复杂的高并发设计。

### 8.1 安全要求

V1 虽然是个人自用系统，但由于直接连接交易所并可能影响真实资金，安全要求不能按普通管理后台来放松。V1 的安全重点不在多用户权限，而在“凭据保护、风险边界、日志脱敏和高风险操作可控”。

系统必须满足以下安全要求：

#### 8.1.1 登录与本地访问安全

- Web 控制台必须要求登录后才能访问核心页面和控制动作。
- 登录凭据必须使用强哈希方式存储，不得以明文或可逆方式保存。
- 控制台必须具备基础会话管理能力，包括会话过期、主动失效或等效安全机制。
- 登录流程必须具备限流能力，防止短时间内被暴力尝试。
- 连续登录失败达到阈值后，系统应触发临时锁定或等效限制。

#### 8.1.2 凭据与敏感配置安全

- OKX API Key、Secret、Passphrase、模型密钥、Telegram 凭据等敏感信息不得硬编码在仓库中。
- 敏感配置必须通过环境变量、密钥管理或等效安全方式提供。
- 模拟盘与实盘的凭据必须严格区分，不得共用同一套配置。
- 系统应明确区分“可展示配置”和“绝不展示配置”，避免在控制台中泄露敏感字段。

#### 8.1.3 日志与数据脱敏

- 普通日志中不得输出完整密钥、完整密码、完整令牌、完整 Cookie 或其他高敏感字段。
- 若业务排障确需打印关键标识，应采用脱敏形式，例如只显示前后少量字符。
- OpenClaw 返回内容也必须遵循同样的脱敏规则，不得因为是内部使用场景而放宽。
- 在错误日志和异常堆栈中，系统应尽量避免把整段敏感配置对象直接序列化输出。

#### 8.1.4 高风险动作保护

- 切换到实盘、强平、手动开仓、修改核心风控策略等动作必须被识别为高风险动作。
- 高风险动作必须要求二次确认，不得一键直接生效。
- 高风险动作必须记录完整审计信息，包括触发时间、触发入口、目标对象、执行结果和关联 ID。
- Web 控制台与 OpenClaw 对高风险动作的边界定义必须一致。

#### 8.1.5 环境隔离与最小暴露

- 模拟盘与实盘的配置、状态、日志和快照应具备明确环境标识，防止串用。
- 实盘状态在界面和日志中应有清晰提示，避免误以为当前仍在模拟盘。
- 当系统处于实盘模式时，任何可能改变风险边界的配置变更都应被额外记录和提示。

### 8.2 可靠性要求

本系统是持续运行型系统，且涉及消息轮询、AI 决策、交易执行和状态同步多个异步链路。V1 的可靠性重点在于：不漏、不错、不重复，以及重启后能尽快回到正确状态。

系统必须满足以下可靠性要求：

#### 8.2.1 去重与幂等

- 同一条未变化的消息不得重复触发决策。
- 同一条决策不得因为重试或重复提交而重复下单。
- 同一待确认项不得被重复确认执行。
- 同一真实订单请求必须具备客户端级幂等标识，例如 `cl_ord_id`。
- 所有关键链路都应能在“同一请求被重复处理一次”的情况下保持结果一致。

#### 8.2.2 重启恢复

- 系统重启后必须能够恢复频道抓取进度，而不是从头重复抓取全部消息。
- 系统重启后必须能够恢复未完成的订单状态、真实持仓状态和账户状态。
- 系统重启后必须能够恢复人工确认队列、虚拟子仓位状态和关键配置状态。
- 如果恢复期间发现本地状态与交易所状态不一致，系统应记录异常并尝试重新对齐。

#### 8.2.3 异常处理与重试

- 对临时性失败，例如网络波动、抓取超时、短时 API 异常，系统应具备受控重试机制。
- 对不可安全重试的动作，例如已经提交但未知结果的下单请求，系统不得盲目重复提交，而应先查单或进行状态确认。
- 抓取失败、AI 调用失败、订单提交失败和状态同步失败都必须分别记录错误上下文。
- 失败重试不应造成无上限堆积，应有合理的退避、上限或熔断策略。

#### 8.2.4 一致性与最终对齐

- 系统必须区分“本地计划状态”和“交易所真实状态”，并以交易所真实状态作为订单和成交的最终依据。
- 虚拟子仓位账本必须以真实成交结果为核心依据进行修正，不得长期停留在纯计划状态。
- 当内部预期与交易所真实结果冲突时，系统必须留下异常记录，并进入补偿或人工关注路径。
- 决策、订单、成交、账本和日志之间必须能通过关联 ID 串起来，避免出现“查到订单但找不到来源消息”的断链情况。

#### 8.2.5 数据保留与可回放

- 原始消息、消息版本、AI 决策、人工确认、真实订单、真实成交和虚拟账本事件都必须保留，不得只保留最终状态。
- 至少在 V1 阶段，账本事件应足够支持问题复盘和状态重建。
- 对关键对象的删除应优先采用逻辑删除或终态标记，而不是物理删除。

### 8.3 可观测性要求

由于本系统包含采集、AI、执行和账本四条核心链路，若没有可观测性，问题会非常难排查。V1 的可观测性目标是让使用者能够快速回答三类问题：

- 现在系统在做什么
- 刚才为什么没做或做错了
- 某笔交易或某条消息最终走到了哪里

系统必须满足以下可观测性要求：

#### 8.3.1 统一日志要求

- 所有关键模块都必须输出结构化或至少规范化日志。
- 关键日志必须包含时间戳、日志级别、模块名和关联 ID。
- 重要业务事件应具备清晰事件名，例如消息抓取、标准化完成、AI 决策完成、确认创建、订单提交、成交同步、账本更新等。
- 日志应支持按照频道、消息、订单、虚拟持仓和关联 ID 进行检索或过滤。

#### 8.3.2 链路追踪能力

- 每条消息处理链路都应生成主 `correlation_id`。
- 同一链路中的日志、决策、确认、订单和账本事件应共享或可推导出同一关联标识。
- 使用者应能够从一条消息追到对应决策、再追到对应订单、再追到对应成交和账本变化。

#### 8.3.3 运行状态展示

- Web 控制台应能实时或准实时展示系统日志。
- 控制台应能展示频道当前抓取状态、最近一次成功时间、最近一次错误摘要。
- 控制台应能展示当前环境、全局交易开关状态、人工确认队列数量、关键任务异常状态。
- OpenClaw topic 应提供摘要级日志输出，帮助在不打开控制台时了解系统近况。

#### 8.3.4 审计可追溯性

- 所有人工确认、高风险操作、环境切换和关键配置变更都必须留有审计记录。
- 审计记录至少应能回答“什么时候、通过什么入口、对什么对象、做了什么、结果怎样”。
- 即使系统只有单一使用者，也必须能区分自动动作与人工动作。

#### 8.3.5 异常可见性

- 抓取失败、AI 调用失败、执行失败、同步失败、恢复失败等异常必须对使用者可见，不能只写底层日志后沉没。
- 控制台中应至少能看到最近错误摘要和当前是否存在待处理异常。
- 对可能影响真实资金的异常，应在日志中有更明显的标记或分类。

### 8.4 性能要求

V1 的性能目标不是追求极限吞吐，而是保证在个人使用场景下，系统能在合理时间内完成“抓取 -> 决策 -> 执行 -> 同步 -> 展示”这一链路，不因为内部阻塞而错过常见频道信号。

系统必须满足以下性能要求：

#### 8.4.1 采集性能

- 默认轮询周期为 `30s` 时，系统应能在一个轮询窗口内完成常规频道抓取任务。
- 单个频道抓取异常不应阻塞整个轮询批次长期卡住。
- 抓取层应避免对同一页面进行明显重复和无意义的高频请求。

#### 8.4.2 决策时效

- 从新消息被确认需要处理，到生成 AI 决策结果，整体时效应满足个人跟单场景下的可用要求。
- 对 `hold`、`ignore` 这类无需执行的结果，也应尽快返回并记录，而不是长时间占用链路。
- 对需要人工确认的新开仓，应尽快进入确认队列，避免因为排队过久而失去时效性。

#### 8.4.3 执行与同步时效

- 对可自动执行的动作，决策完成后应尽快进入执行规划与下单流程。
- 订单提交后，系统应尽快同步真实订单和成交状态，避免控制台长期显示过时信息。
- 虚拟子仓位和 PnL 展示应在关键成交回报后尽快刷新。

#### 8.4.4 控制台体验性能

- 控制台日志、持仓、挂单和确认队列应尽量实时刷新或以接近实时的方式展示。
- 常用页面在个人日常使用场景下应保持流畅，不应因为数据量增长而立即变得不可用。
- 即使日志量增加，控制台也应优先保证最近关键状态可查看，而不是一次性加载全部历史数据。

#### 8.4.5 性能边界说明

- V1 不需要按超高并发、多租户或海量频道运营场景设计。
- 但 V1 必须避免明显的串行瓶颈、重复处理和无边界阻塞，否则即使在单人使用场景下也会影响实用性。
- 后续若频道数量、消息频率或模型调用量显著上升，可在 V1.1 或后续版本中再细化容量规划与压测标准。

## 9. 配置项与默认值

本章用于明确系统有哪些配置、哪些值已经被确认为 V1 默认值、哪些配置允许在运行时修改，以及哪些配置必须严格区分模拟盘与实盘。该章节的目标不是要求首版控制台一次性把所有配置都做成可视化页面，而是先明确配置边界，避免实现时出现“哪些该做成设置，哪些不该让人乱改”这种分歧。

V1 的配置设计应遵循以下原则：

- 已确认的业务边界应固化为默认值或明确规则，不能在实现中被悄悄改掉。
- 机密配置与业务配置必须分开管理。
- 运行时配置应尽量支持热更新，但高风险配置必须伴随确认和审计。
- 模拟盘与实盘相关配置必须隔离，不能共享为一套模糊不清的数据。

### 9.1 默认配置

以下默认值属于当前已经确认的 V1 起始配置或固定业务默认规则。在没有后续明确需求变更前，系统应按以下口径实现。

#### 9.1.1 接入与采集默认值

- Telegram 默认来源为网页抓取，即 `telegram_web` 或等效来源类型。
- 默认抓取频率为 `30s`。
- 默认只处理公开频道，不要求 V1 支持 Bot API 或 TDLib 接入。

#### 9.1.2 交易范围默认值

- 默认交易所为 `OKX`。
- 默认交易品类为 `合约 / 永续合约`。
- 默认账户范围为单账户。
- 默认持仓模式为 `单向`。
- 默认保证金模式为 `逐仓`。
- 默认杠杆为 `25x`。

#### 9.1.3 执行策略默认值

- 默认订单类型为 `限价`。
- 当频道明确信号要求市价时，允许使用市价单。
- 默认未成交订单超时时间为 `5h`。
- 对 OKX 未上线标的的默认处理方式为 `跳过`。
- 频道隔离的默认实现方式为 `虚拟子仓位账本`。

#### 9.1.4 决策与确认默认值

- 新开仓低置信度的默认处理方式为 `人工确认`。
- 已有持仓的跟进动作默认不因低置信度而被阻塞。
- 新开仓资金投入比例默认遵循“在可用资金 `40%-80%` 范围内由 AI 决策”的规则。
- AI 上下文窗口默认推荐值为“同频道最近 `8` 条相关消息”，但允许手动调整。

#### 9.1.5 控制台安全默认值

- Web 控制台采用单用户模式。
- 登录认证默认采用 `6` 位密码方案。
- 登录安全默认包含限流、失败锁定、会话超时和操作审计。

#### 9.1.6 默认值说明

除上述已确认默认值外，以下事项不应在实现中依赖“隐式猜测值”：

- 当前运行环境是模拟盘还是实盘
- 全局交易开关初始状态
- 高风险动作确认开关

这类直接影响风险边界的配置，应要求在部署或首次初始化时明确设置，并在界面中清楚展示当前值。

### 9.2 可配置项

V1 的可配置项可分为部署级机密配置、运行时业务配置和系统维护状态三类。其中真正面向使用者开放的，主要是运行时业务配置；机密配置应通过更安全的方式提供；系统维护状态则主要由系统自动写入，不应作为普通设置项随意编辑。

#### 9.2.1 部署级机密配置

以下配置属于敏感配置，不应在普通控制台中以明文形式编辑或展示：

- OKX 模拟盘 API Key、Secret、Passphrase
- OKX 实盘 API Key、Secret、Passphrase
- OpenAI 或其他模型服务密钥
- Telegram 相关凭据或受保护的接入令牌
- 其他第三方服务访问凭据

这类配置必须满足以下要求：

- 应通过环境变量、密钥管理或等效安全方式注入。
- 模拟盘和实盘凭据必须分组保存。
- 控制台最多只展示“是否已配置”和部分脱敏摘要，不展示完整值。
- 修改这类配置后，系统应记录发生过配置变更，但不应在审计记录中写入明文内容。

#### 9.2.2 频道与采集配置

以下配置应属于运行时业务配置，可保存在数据库或等效配置存储中：

- 频道列表
- 频道启停状态
- 频道展示名称和备注
- 频道来源地址或抓取标识
- 频道级抓取参数
- 频道级自动执行开关或等效控制项

这类配置必须满足以下要求：

- 新增、编辑、停用应支持热更新。
- 频道配置修改不应清空其历史消息、历史决策和历史账本。
- 频道停用后至少应阻止新消息继续进入自动执行链路。

#### 9.2.3 AI 与决策配置

以下配置应作为运行时可调整的业务配置：

- 当前使用的模型
- 当前思考等级
- AI 上下文窗口长度
- 新开仓人工确认阈值
- 动作白名单或允许动作范围
- AI 输出结构化约束的版本或规则配置

这类配置必须满足以下要求：

- 模型和思考等级切换应尽量对后续新决策即时生效。
- AI 上下文窗口长度调整应只影响后续新决策，不回溯重算历史结果。
- 配置变更只影响变更后的新决策，不要求回溯重算历史决策。
- 人工确认阈值属于高影响配置，变更后必须有审计记录。

#### 9.2.4 交易执行与风险配置

以下配置应被视为高重要度业务配置：

- 当前运行环境：模拟盘或实盘
- 全局交易开关
- 默认杠杆
- 新开仓资金投入比例范围
- 订单超时撤销时间
- 允许的执行动作范围
- 高风险动作的确认策略

这类配置必须满足以下要求：

- 环境切换必须视为高风险动作。
- 全局交易开关关闭时，系统应继续采集和决策，但停止真实执行。
- 默认杠杆和资金投入比例范围应只影响新生成的执行动作，不回溯改写历史订单。
- 对“投入比例范围”这类高风险策略，V1 默认采用 `40%-80%` 规则；若后续要突破该边界，应视为需求变更而非普通日常调参。
- 订单超时撤销时间可作为系统策略配置保存，但首版不强制要求一定在控制台中开放为独立页面项。

#### 9.2.5 控制台与安全配置

以下配置属于单用户控制台层配置：

- 登录密码
- 会话超时时间或等效策略
- 登录失败锁定阈值
- 限流策略参数
- 控制台展示偏好或刷新策略（若后续实现）

这类配置必须满足以下要求：

- 登录密码修改必须要求重新验证或等效安全确认。
- 安全相关配置的修改必须写入审计记录。
- 首版控制台不需要提供多用户、多角色或权限矩阵类配置。

#### 9.2.6 OpenClaw 配置

以下配置可作为 OpenClaw 集成相关配置：

- 日志输出目标 topic 或等效目标位置
- 允许通过 OpenClaw 执行的低风险动作清单
- OpenClaw 查询能力开关或返回范围限制

这类配置必须满足以下要求：

- OpenClaw 可执行动作必须严格受白名单控制。
- 高风险动作不得因为配置疏漏而被默认放开。
- OpenClaw 配置修改也应保留审计记录。

#### 9.2.7 系统维护状态与非人工配置项

以下内容虽然会被保存，但更偏向系统维护状态，而不是普通“用户设置”：

- 抓取 cursor
- 最近错误摘要
- 最近成功时间
- 账户快照与持仓快照
- 消息处理状态
- 确认队列状态

这类信息应满足以下要求：

- 由系统自动维护，不应在普通设置页中开放手工任意编辑。
- 若确实需要人工修正，应视为运维级动作并写入审计记录。

### 9.3 配置生效与隔离规则

除“有哪些配置”之外，V1 还必须明确配置是如何生效、如何隔离以及如何审计的。

#### 9.3.1 生效规则

- 频道启停、模型切换、思考等级切换、全局交易开关等运行时配置，应尽量支持热更新。
- 配置变更默认只影响变更后的新处理链路，不追溯改写已生成的历史记录。
- 若某项配置需要重启或延迟生效，系统必须明确提示，而不能让使用者误以为已经立即生效。

#### 9.3.2 环境隔离规则

- 模拟盘与实盘的凭据、订单、成交、快照和关键配置必须能明确区分。
- 控制台中必须明确显示当前环境，避免误操作。
- 与实盘有关的配置和状态不得因为读取默认值错误而回落到模拟盘，反之亦然。

#### 9.3.3 审计规则

- 所有关键配置变更都必须写入审计记录。
- 至少应记录变更时间、变更入口、目标配置项、变更结果和关联 ID。
- 对敏感配置和高风险配置，应记录“发生过变更”这一事实，但不记录完整明文值。

#### 9.3.4 配置边界规则

- 已确认为 V1 固定业务规则的内容，不应被误做成可自由修改项。
- 首版控制台不需要把所有配置都做成页面开关，但后端配置模型应预留扩展空间。
- 凡是会直接改变风险边界的配置，默认都应归入高优先级审计范围。

## 10. 验收标准

本章用于定义 V1 是否达到“可交付、可自用、可继续进入后续实现和联调阶段”的验收口径。验收标准的目标不是追求完美，而是确保前文已经确认的范围边界都能在系统中被真实体现，而不是只停留在文档描述上。

V1 的验收应遵循以下原则：

- 先验收闭环是否跑通，再验收边界是否守住。
- 先验收模拟盘能力，再决定是否进入实盘准备。
- 验收必须以可观察结果为准，而不是只看代码存在与否。
- 未纳入 V1 范围的能力，不应作为 V1 未通过的原因。

### 10.1 功能验收

当以下功能项全部达到预期时，可认为 V1 在功能层面达到基本验收标准。

#### 10.1.1 频道与采集能力验收

以下条件全部成立时，该部分视为通过：

- 使用者可以通过 Web 控制台登录并查看频道列表。
- 使用者可以在不重启服务的情况下新增、修改、停用和删除监控频道。
- 新增或修改频道后，系统能够按新的配置继续进行监控。
- 系统能够基于网页抓取方式识别 Telegram 公开频道中的新消息。
- 系统能够识别消息编辑，并保存旧版本与新版本关系。
- 抓取异常不会导致整个系统停止处理其他频道。

判定口径如下：

- 至少能用一个真实公开频道完成从“新增频道 -> 开始抓取 -> 看到消息入库”的闭环。
- 编辑消息后，系统中能看到版本变化，而不是只覆盖原文本。
- 频道停用后，新消息不再继续进入自动处理链路。

#### 10.1.2 AI 决策能力验收

以下条件全部成立时，该部分视为通过：

- 系统能够将单条消息和同频道上下文一起发送给 AI 决策模块。
- AI 决策结果能够以结构化格式保存，而不是不可解析的自由文本。
- 结构化结果至少包含标的、方向、动作类型、关键价格信息、置信度和简短理由。
- 系统能够区分新开仓与跟进动作。
- 对低置信度新开仓，系统能够进入人工确认队列。
- 对不应执行的场景，系统能够输出 `hold` 或 `ignore` 等非执行结果。

判定口径如下：

- 至少选取若干条不同类型样本消息，验证系统能稳定产出结构化决策记录。
- 同一频道上下文应只包含本频道消息，不得混入其他频道数据。
- 当消息发生编辑后，系统能够基于新版本重新生成新决策记录。

#### 10.1.3 人工确认能力验收

以下条件全部成立时，该部分视为通过：

- 低置信度新开仓能够生成待确认项。
- 控制台中能够查看待确认项详情，包括消息原文、上下文摘要、AI 决策结果和关键参数。
- 使用者能够确认执行或拒绝执行。
- 拒绝执行后，系统不会继续自动执行该动作。
- 同一待确认项不会因为重复点击而重复执行。

判定口径如下：

- 至少完成一次“生成待确认项 -> 人工确认 -> 进入执行”的完整链路。
- 至少完成一次“生成待确认项 -> 人工拒绝 -> 不再执行”的完整链路。
- 若确认项在等待期间已失效，系统应能识别并阻止旧确认继续执行。

#### 10.1.4 OKX 模拟盘执行能力验收

以下条件全部成立时，该部分视为通过：

- 系统能够连接 OKX 模拟盘环境。
- 系统能够查询标的、余额、持仓和挂单状态。
- 系统能够提交合约相关订单，并同步订单状态。
- 系统能够执行撤单、改单、止损、止盈、多个止盈点、部分止盈和保本移动等管理动作。
- 系统能够对未上线标的执行跳过逻辑，而不是报错中断主链路。
- 系统能够对未成交订单在超时后执行自动撤销。

判定口径如下：

- 至少在模拟盘完成一次开仓、一次撤单或改单、一次止盈止损管理类动作。
- 订单提交后，控制台中能够看到真实订单状态变化。
- 未成交超时单的自动撤销在规则触发后能够被日志和状态记录观察到。

#### 10.1.5 真实状态同步与虚拟账本验收

以下条件全部成立时，该部分视为通过：

- 系统能够同步真实订单、真实成交、真实持仓和账户余额变化。
- 系统能够维护频道级虚拟子仓位，而不是只展示交易所真实净仓位。
- 虚拟子仓位能够记录生命周期、虚拟均价、事件流和盈亏估算。
- 当多个频道作用于同一标的时，系统仍能区分各频道自己的虚拟状态。
- 跟进消息触发的管理动作能够回写到账本与审计链路中。

判定口径如下：

- 至少验证一次“真实成交发生后，虚拟子仓位同步更新”的链路。
- 至少验证一次“同一标的在不同频道下保留独立虚拟状态”的场景。
- 从任一条账本事件能够追溯到来源消息、决策和真实订单映射。

#### 10.1.6 Web 控制台验收

以下条件全部成立时，该部分视为通过：

- 控制台能够完成单用户登录。
- 控制台能够展示频道、日志、订单、持仓、虚拟子仓位和 PnL 信息。
- 控制台能够展示当前运行环境、全局交易开关状态和关键异常摘要。
- 控制台能够处理人工确认项。
- 控制台能够执行频道启停、模型切换、思考等级切换和全局交易开关等基础操作。

判定口径如下：

- 使用者能够在一个控制台中完成“查看状态 -> 调整配置 -> 处理确认”的连续操作。
- 控制台中的关键状态与系统后台实际状态保持一致，不出现长时间明显错位。

#### 10.1.7 OpenClaw 集成验收

以下条件全部成立时，该部分视为通过：

- OpenClaw 能在 Telegram topic 中输出系统日志摘要。
- OpenClaw 能响应基础状态查询。
- OpenClaw 能执行已允许的低风险动作，例如暂停频道、恢复频道。
- OpenClaw 对高风险动作不会直接放行，而是进入确认或拒绝路径。

判定口径如下：

- 至少完成一次通过 OpenClaw 查询系统状态。
- 至少完成一次通过 OpenClaw 执行低风险控制动作。
- 至少验证一次高风险动作被要求确认或被阻止的行为。

### 10.2 安全与运维验收

当以下安全与运维项全部达到预期时，可认为 V1 具备进入长期运行和后续上线准备的基础条件。

#### 10.2.1 安全验收

以下条件全部成立时，该部分视为通过：

- 控制台登录密码未以明文或可逆形式保存。
- 敏感凭据未出现在前端展示中。
- 敏感凭据未出现在普通日志和 OpenClaw 返回内容中。
- 高风险动作具备二次确认和审计记录。
- 模拟盘与实盘凭据和状态能够明确区分。

判定口径如下：

- 通过日志抽查、界面抽查和配置抽查，确认无敏感信息泄露。
- 通过高风险动作演示，确认系统不会跳过确认直接执行。

#### 10.2.2 可靠性验收

以下条件全部成立时，该部分视为通过：

- 消息重复抓取不会导致重复决策或重复下单。
- 系统重启后能够恢复抓取进度、未完成订单状态和关键配置状态。
- 重启后系统能够继续同步交易所状态，而不是进入永久失联状态。
- 已存在的人工确认项、虚拟子仓位和日志记录不会因重启而丢失。

判定口径如下：

- 至少验证一次重复消息场景下未产生重复执行。
- 至少验证一次服务重启后系统继续从正确位置恢复运行。

#### 10.2.3 可观测性与审计验收

以下条件全部成立时，该部分视为通过：

- 关键业务事件能够在日志中被看到。
- 日志中能够通过关联 ID 追踪一条完整链路。
- 人工确认、配置变更、环境切换和高风险动作都有审计记录。
- 控制台能看到最近错误摘要和关键运行状态。

判定口径如下：

- 至少随机抽取一条消息处理链路，验证可从日志追到决策、订单和账本变化。
- 至少随机抽取一次人工操作，验证其审计记录完整。

#### 10.2.4 V1 达标结论

当以下条件同时成立时，可认定 V1 达到当前阶段验收目标：

- 核心功能链路已跑通：采集、标准化、上下文构建、AI 决策、人工确认、OKX 模拟盘执行、状态同步、虚拟账本、控制台、OpenClaw。
- 已确认的业务边界已被守住：单用户、自用系统、仅网页抓取、仅 OKX、仅合约 / 永续、仅单账户、仅单向持仓、仅新开仓低置信度进入人工确认。
- 系统具备基本安全、恢复、审计和可观测能力。
- 系统能够在模拟盘环境下稳定完成日常使用所需的核心闭环。

若以上任一核心项未通过，则 V1 仍应视为“草稿完成但未验收通过”，不建议进入真实资金使用阶段。

## 11. 实施优先级与交付阶段

本章用于说明 V1 的实际实施顺序。虽然系统从技术上看是“采集 -> AI -> 执行 -> 控制台”的链路，但基于当前项目目标和使用方式，交付顺序不应完全按照技术依赖线性展开，而应优先锁定 Web 控制台的视觉方向、信息架构和交互框架。原因很明确：本系统最终会长期由你自己使用，而你对前端审美和界面质感要求很高，因此前端需要尽早定型，避免后面后端都做完了再返工界面结构。

### 11.1 排期原则

V1 的排期应遵循以下原则：

- 先做 Web 前端框架和视觉原型，尽早确认信息架构、页面层级和视觉风格。
- 再补后端基础骨架和认证配置，让前端尽快接上真实数据和真实操作。
- 交易执行相关能力先以模拟盘打通，实盘只做准备和边界控制，不作为首批联调目标。
- 高风险、低频、附加值较低的能力放到后面，例如 OpenClaw 深化和上线前强化。

### 11.2 Phase 1：Web 前端优先

第一阶段优先完成 Web 控制台的可视化框架和视觉方向确认。该阶段的目标不是先把所有后端能力做完，而是先把“这个系统以后每天怎么用、看起来是否顺手、是否符合审美要求”定下来。

本阶段建议交付内容如下：

- 控制台整体视觉方向、色彩系统、排版和布局规范
- 登录页和主应用骨架
- 首页 / 仪表盘信息架构
- 频道管理页原型
- 日志页原型
- 持仓 / 挂单 / PnL 页原型
- 人工确认页原型
- 关键状态栏设计，例如当前环境、全局交易开关、系统健康状态

本阶段通过标准如下：

- 主要页面结构和交互路径已经定稿
- 视觉风格达到可接受水平
- 后续后端接口接入时不需要再大改页面框架

### 11.3 Phase 2：后端基础骨架与控制台接线

第二阶段在前端框架确定后，补齐后端最小骨架，让前端可以开始接真实配置和基础状态。

本阶段建议交付内容如下：

- 后端项目初始化
- 数据库与基础配置系统
- 单用户登录认证
- 会话管理、限流、失败锁定
- 基础设置接口
- 基础日志接口或实时推送通道
- 前后端基础联调

### 11.4 Phase 3：Telegram 接入与消息链路

第三阶段完成消息进入系统的主入口，让控制台能看到真实频道数据。

本阶段建议交付内容如下：

- Telegram 网页抓取
- 抓取 cursor 管理
- 原始消息保存
- 消息标准化
- 编辑消息识别
- 频道配置热更新

### 11.5 Phase 4：AI 决策与人工确认

第四阶段完成消息理解能力和人工确认闭环。

本阶段建议交付内容如下：

- 上下文构建
- AI 结构化输出
- 模型切换
- 思考等级切换
- 新开仓人工确认队列
- 决策记录落库与控制台展示

### 11.6 Phase 5：OKX 模拟盘执行与状态同步

第五阶段开始打通交易执行闭环，但以模拟盘为主。

本阶段建议交付内容如下：

- OKX 模拟盘连接
- 杠杆设置、查余额、查持仓、下单、撤单、改单
- 止损、止盈、多个止盈点、部分止盈、保本移动
- 订单状态同步
- 成交状态同步
- 持仓与账户状态同步

### 11.7 Phase 6：虚拟子仓位账本与 PnL

第六阶段补齐系统最核心的频道级账本能力。

本阶段建议交付内容如下：

- 频道级虚拟子仓位主记录
- 虚拟持仓事件流
- 真实订单与虚拟动作映射
- 已实现 / 未实现盈亏估算
- 控制台中的虚拟持仓与 PnL 面板联动

### 11.8 Phase 7：OpenClaw 集成

第七阶段补齐 Telegram 协作侧的轻量入口。

本阶段建议交付内容如下：

- 日志 topic 输出
- 基础状态查询
- 低风险控制动作
- 高风险动作确认边界
- 项目专用 skill

### 11.9 Phase 8：强化与上线准备

最后阶段用于做真实长期运行前的补强，而不是继续扩范围。

本阶段建议交付内容如下：

- 去重、幂等和恢复能力强化
- 安全检查与脱敏检查
- 审计链路补全
- 异常处理和重试策略完善
- 实盘切换保护与演练
- 验收清单逐项回归
